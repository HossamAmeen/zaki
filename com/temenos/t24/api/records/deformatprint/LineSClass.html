        public TField getDependentOn()
        Returns the DependentOn

        Field description: Specifies that the FIELD/"TEXT" (Field 8.1) is not to be printed if the field named in DEPENDENT ON is blank or zero, or is only printed if the field named in DEPENDENT ON matches the condition specified. This field is generally used to prevent a heading from being printed when the associated data is not present. Example: FIELD/"TEXT" "PAYMENT DETAILS" DEPENDENT ON PAY DET ... FIELD/"TEXT" PAY DET The heading PAYMENT DETAILS is only needed if the field PAY DET contains something to be printed. Example: FIELD/"TEXT" AMOUNT CALCULATION +,TOTAL.1 DEPENDENT ON AMOUNT FIELD/"TEXT" "--------------" DEPENDENT ON TOTAL.1 FIELD/"TEXT" TOTAL DEBIT DEPENDENT ON In the above example TOTAL.1 is not printed. It is only used in the Dependent On field to avoid drawing a line across if there are no detail lines. AMOUNT will not print if it contains zero because it is 'Dependent On' itself. The 'Dependent On' parameter suppresses both blank and zero values. TOTAL DEBIT will print if it is zero. In addition, a field or text can be printed depending on the value of the 'DEPENDENT ON' field. For example, print an amount in a credit or debit column according to the value of the amount. LINE 10 COLUMN 35 FIELD/"TEXT" AMOUNT DEPENDENT ON AMOUNT DEPEND OPERAND LT DEPEND COND 0 LINE 10 COLUMN 50 FIELD/"TEXT" AMOUNT DEPENDENT ON AMOUNT DEPEND OPERAND GE DEPEND COND 0 In the above example, if the amount is less than 0, it will be printed in column 35. If, however, it is greater than or equal to 0, it will be printed in column 50. Fields can only be compared to constants (and not to other fields). The fields compared against (i.e. the DEPENDENT ON field) is the field as it was passed by the application. If the field is dependent upon itself, the value of the field currently being processed is compared against. The appropriate value of the field is used if the field is multivalued. However, if a multivalue field can be printed on subsequent pages of the report, it is not possible to know the current value of the field. Therefore, a field which is printed in the format can be prefixed with a '*' to compare the current value of the field. If printing a field dependent on a field prefixed with an asterisk, it should be noted that DEPENDENT ON field is cleared down after it is printed. Therefore, if the FIELD/"TEXT" to be printed is dependent on field which has already been printed, the DEPENDENT ON field will be null. Example using '*' field LINE 10-25 COLUMN 10 MULTIVALUE M FIELD/"TEXT" AMOUNT DEPENDENT ON AMOUNT PAGE OVERFLOW Y LINE 10-25 COLUMN 20 MULTIVALUE M FIELD/"TEXT" AMOUNT PAGE OVERFLOW Y In the above example, the text 'amount' is printed if the current value of AMOUNT is present. If the DEPENDENT ON field were specified without and asterisk the checking would not work on the second and subsequent pages. Validation Rules: The name of any field in the Message, (it need not be one that is being printed), or the name of any field in the Format record prefixed by '*' (i.e. it must be a field that is being printed) or TOTAL.n, where n = 1 to 9 (Optional input)

        public TField getIndent()
        Returns the Indent

        Field description: Specifies the positioning of a field across a page, expressed as character position across a print line (starting from 1). Unless a Mask is specified printing will be left justified so that it begins printing at the point specified in INDENT. Fields to be printed must all be defined in sequence down a page, but fields may be superimposed across a line so that when one is not present another will show. When a field is positioned on a print line it starts from the character position defined by Indent then, regardless of the length of the field, the gap between the end of the field and the end of the print line is set to blanks. This has the advantage that a short field may optionally be used to overwrite a long field, but it does mean that variables cannot be inserted into text. Therefore, in this example, three fields must be defined: "Interest will be on a" followed by the field INTEREST BASIS e.g. 365 followed by "day basis." Validation Rules: 1-3 numeric characters. (Mandatory input) INDENT must not exceed the width of the FORM.TYPE specified for this format.

      • getLineS

        public TField getLineS()
        Returns the LineS

        Field description: Specifies how far down a page an item defined in FIELD/"TEXT" (Field 8.1) should be printed. It gives either an explicit line number or the line number relative to the last item printed. Additionally it is possible to specify the highest line number that this item may be printed on or (for multi- or subvalued fields) the maximum number of lines to print. This field contains: 1) Either: a) The line number (counting from 1 at the top of the page) on which to print. Example: 1 (Print at the top of the page.) or b) The relative line position: the number of lines to move down the page from the last item printed. Example: +2 (Miss 1 line after the last item then print on next line.) 2) '-' 3) Either: a) The highest line number to be used. Example: +1-55 (Print on the next line, but not if line is 56 or higher.) or b) '+' and the maximum number of lines to be printed. Example: +0-+5 (Print on the same line as the last item, then print subsequent multi- or subvalues on the following lines, but do not print more than 5 lines.) Part 1 is always required but parts 2 and 3 which are specified together are optional. Note: When part 1 b) in the above details is used it differs from part 3 b) as follows: '+' before the first number means that it specifies a displacement of the given number of lines from the previous field printed. When the second number is preceded by '+' it specifies the maximum number of lines to be printed, not the maximum displacement from the previous field position. Examples: 02 Print on line 2. +2 Print 2 lines down from the last item printed. (Miss 1 line.) +0 Print on the same line as the last item. 1-10 Start printing on line 1, put the next multivalue on line 2, ... put the 10th value on line 10, if that many exist. If more than 10, page overflow will occur. +2-55 Start printing 2 lines down from the last item printed and do not print past line 55. If the previous item was printed on line 54 or more, this item will overflow immediately onto the next page. +4-+2 Start printing 4 lines down from the last item printed and do not print more than 2 lines. +2-+4 then +0-+4 for the next item. This can be used to print a single or multivalue item alongside a 4-line multivalue set but they must specify the same number of lines. +1 then +1 then +0. If the second item has a Dependent On field this causes it not to print, and the third item will still print one line down from the first item. Any zero displacements and the preceding non-zero Line are regarded as a set. If the number of multivalues or subvalues in a field exceeds the number of lines specified then page overflow is required. If the PAGE OVERFLOW (Field 13.1) is set to 'NO' the message will be placed on the Repair Queue. Once an item has overflowed nothing else with a relative line position will be printed on that page unless it has 'Heading' set to 'B' or 'S'. Example: LINE(S) 3-55 FIELD/"TEXT" AMOUNT LINE(S) +2 FIELD/"TEXT" TOTAL If AMOUNT needs to go beyond line 55 page overflow will occur. TOTAL, which has a 'relative' line specification will not print until all the values of AMOUNT have printed, even though TOTAL is allowed to print past line 55. Validation Rules: a) '+' (Optional) b) 1-2 numeric characters (Mandatory) c) '-' (Optional) d) '+' (Optional with part c, otherwise not allowed.) e) 1-2 numeric characters (Mandatory with part c, otherwise not allowed.) Fields must be defined sequentially down a page. Fields printed alongside each other must have the same number of lines specified, even if one is a single value and the other a multivalue set. The single value will need to have a range of lines specified, but 'MULTI' will be left blank so that it only prints once. Line number must not be greater than the form depth specified in DE.FORM.TYPE file.

        public void setIndent(TField value)
        Set the specified value for Indent
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Specifies the positioning of a field across a page, expressed as character position across a print line (starting from 1). Unless a Mask is specified printing will be left justified so that it begins printing at the point specified in INDENT. Fields to be printed must all be defined in sequence down a page, but fields may be superimposed across a line so that when one is not present another will show. When a field is positioned on a print line it starts from the character position defined by Indent then, regardless of the length of the field, the gap between the end of the field and the end of the print line is set to blanks. This has the advantage that a short field may optionally be used to overwrite a long field, but it does mean that variables cannot be inserted into text. Example: Suppose the following text has been defined: "Interest will be on a XXXXXXX day basis." XXXXXXX cannot be successfully overwritten with the appropriate field name, such as INTEREST BASIS, because in the above example the words 'day basis' would be blanked out when the INTEREST BASIS field was put in. Therefore, in this example, three fields must be defined: "Interest will be on a" followed by the field INTEREST BASIS e.g. 365 followed by "day basis." Validation Rules: 1-3 numeric characters. (Mandatory input) INDENT must not exceed the width of the FORM.TYPE specified for this format.

        public void setLineS(java.lang.CharSequence value)
        Set the specified value for LineS
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Specifies how far down a page an item defined in FIELD/"TEXT" (Field 8.1) should be printed. It gives either an explicit line number or the line number relative to the last item printed. Additionally it is possible to specify the highest line number that this item may be printed on or (for multi- or subvalued fields) the maximum number of lines to print. This field contains: 1) Either: a) The line number (counting from 1 at the top of the page) on which to print. Example: 1 (Print at the top of the page.) or b) The relative line position: the number of lines to move down the page from the last item printed. Example: +2 (Miss 1 line after the last item then print on next line.) 2) '-' 3) Either: a) The highest line number to be used. Example: +1-55 (Print on the next line, but not if line is 56 or higher.) or b) '+' and the maximum number of lines to be printed. Example: +0-+5 (Print on the same line as the last item, then print subsequent multi- or subvalues on the following lines, but do not print more than 5 lines.) Part 1 is always required but parts 2 and 3 which are specified together are optional. Note: When part 1 b) in the above details is used it differs from part 3 b) as follows: '+' before the first number means that it specifies a displacement of the given number of lines from the previous field printed. When the second number is preceded by '+' it specifies the maximum number of lines to be printed, not the maximum displacement from the previous field position. Examples: 02 Print on line 2. +2 Print 2 lines down from the last item printed. (Miss 1 line.) +0 Print on the same line as the last item. 1-10 Start printing on line 1, put the next multivalue on line 2, ... put the 10th value on line 10, if that many exist. If more than 10, page overflow will occur. +2-55 Start printing 2 lines down from the last item printed and do not print past line 55. If the previous item was printed on line 54 or more, this item will overflow immediately onto the next page. +4-+2 Start printing 4 lines down from the last item printed and do not print more than 2 lines. +2-+4 then +0-+4 for the next item. This can be used to print a single or multivalue item alongside a 4-line multivalue set but they must specify the same number of lines. +1 then +1 then +0. If the second item has a Dependent On field this causes it not to print, and the third item will still print one line down from the first item. Any zero displacements and the preceding non-zero Line are regarded as a set. If the number of multivalues or subvalues in a field exceeds the number of lines specified then page overflow is required. If the PAGE OVERFLOW (Field 13.1) is set to 'NO' the message will be placed on the Repair Queue. Once an item has overflowed nothing else with a relative line position will be printed on that page unless it has 'Heading' set to 'B' or 'S'. Example: LINE(S) 3-55 FIELD/"TEXT" AMOUNT LINE(S) +2 FIELD/"TEXT" TOTAL If AMOUNT needs to go beyond line 55 page overflow will occur. TOTAL, which has a 'relative' line specification will not print until all the values of AMOUNT have printed, even though TOTAL is allowed to print past line 55. Validation Rules: a) '+' (Optional) b) 1-2 numeric characters (Mandatory) c) '-' (Optional) d) '+' (Optional with part c, otherwise not allowed.) e) 1-2 numeric characters (Mandatory with part c, otherwise not allowed.) Fields must be defined sequentially down a page. Fields printed alongside each other must have the same number of lines specified, even if one is a single value and the other a multivalue set. The single value will need to have a range of lines specified, but 'MULTI' will be left blank so that it only prints once. Line number must not be greater than the form depth specified in DE.FORM.TYPE file.
