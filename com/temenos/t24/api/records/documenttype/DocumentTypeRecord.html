        public void addEquivDoc(java.lang.CharSequence param)
        Appends the specified EquivDoc to the end of the collection.
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Specifies document type(s) which are considered as equivalent to this document type, in terms of purpose and processing. If an equivalent document type exists with a valid status, it would be treated for the purposes of document tracking as if the main document type exists with a valid status. Equivalence of document types would be checked at only one level. i.e. If Document Type A has an equivalent Document Type B and Document Type B has an equivalent Document Type C, when A is tracked only the existence of B with a valid status would be checked and any status of C would be ignored.       • getNewAllCust

        public TField getNewAllCust()
        Returns the NewAllCust

        Field description: Specifies whether the document type (if required for Customer creation), needs to be tracked only for new Customers created on or after the BEGIN.DATE. If the value of this field is YES: Document type would be tracked for new customers created on or after the BEGIN.DATE, provided it is a required document type for them. When an existing Customer is modified, and if this document type is a required one, it would be tracked (its Status would be tested for processing) only if it has been already tracked (record exists in the file CUST.DOCUMENT) for that Customer. Validation Rules: Optional input with allowed values YES/NO. A Null value would be interpreted as NO. YES not allowed if the field CONTRACT.SPECIFIC is YES. YES not allowed when equivalent documents (Field: EQUIV.DOC) are specified. This Document Type could be specified as a required document type only for a Customer document group.

        Returns:

        public void setValidPeriod(java.lang.CharSequence value)
        Set the specified value for ValidPeriod
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Specifies the period up to which a Document of this type would be valid from the date specified in the field VALID.BASE.DATE. The validity period could be specified either from the date specified in the field VALID.BASE.DATE or from its calendar year beginning. If the status attached to a document of this type is invalid, then T24 would not calculate its expiry date, using the definition of this field. Refer to the help text for the field END.DATE in the file CUST.DOCUMENT for calculation of expiry date using this field. Example VALID.PERIOD=Null, No End date would be calculated. VALID.PERIOD= 12M, VALID.BASE.DATE=STATUS.DATE. For a document of this type with a STATUS.DATE of 20.2.2002, its End date would be calculated as 19.2.2003. VALID.PERIOD= 12MS, VALID.BASE.DATE=STATUS.DATE - calculation to be done from year Beginning. For a document of this type with a STATUS.DATE of 20.2.2002, its End date would be calculated as 31.12.2002. VALID.PERIOD= 12M, VALID.BASE.DATE=SIG.DATE. For a document of this type with a SIG.DATE of 19.2.2002, its End date would be calculated as 18.2.2003. VALID.PERIOD= 12MS, VALID.BASE.DATE=SIG.DATE - calculation to done from year beginning. For a document of this type with a SIG.DATE of 19.2.2002, its End date would be calculated as 31.12.2002. Validation Rule Optional input. Must be in the format xxxD, xxxM or xxxY to specify the number of days, months or years respectively from the VALID.BASE.DATE. Must be in the format xxxDS, xxxMS or xxxYS to specify the number of days, months or years respectively from the VALD.BASE.DATE's calendar year beginning.

        Parameters:
        value - The value to set
        Returns:
        nothing
