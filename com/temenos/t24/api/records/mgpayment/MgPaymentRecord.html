        public void addOverride(java.lang.CharSequence param)
        Appends the specified Override to the end of the collection.
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Contains all the Override messages which the user agreed to during Input. During the validation of a payment the system may provide the user with a series of screen override messages to indicate an anomaly; for instance, if the value date of the payment entered is not a working day. In the event the user confirming the Override, the full text of the actual Override Message will be stored for reference purposes only in this multivalue field. In the example given above when the deal is validated a warning bell will sound and a message will appear, at the foot of the screen, which reads: 'VALUE DATE IS NOT A WORKING DAY' This will draw the user's attention to the amount entered, which may be a mistake and can then be corrected, or, on the other hand, it may be correct and can be confirmed as so. A second example might be where the payment amount debiting the payment account causes this account to be overdrawn. In this case, when the deal is validated, a warning bell will sound and a message will appear, at the foot of the screen, which reads: 'ACCOUNT 0221000178 - UNAUTHORISED OVERDRAFT' Once again the user's attention is drawn to the situation and either corrected or confirmed via a positive response to the Override Message. Validation Rules: Internal field. Multivalue field.

      • addStmtNo

        public void addStmtNo(java.lang.CharSequence param)
        Appends the specified StmtNo to the end of the collection.
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: A number generated automatically by the system which identifies either a statement entry or a category entry in respect of a contract. Whenever a payment is entered via this application a number of accounting entries will be raised. This field will record the IDs of the entries generated. The IDs to accounting entries, both STMT.ENTRYs and CATEG.ENTRYs, consist of a date-time stamp and a sequence number. All entries raised by a particular input/transaction will have the same date-time stamp but different sequence numbers. This field has three multivalues, the first contains the base date-time stamp which is common to all entries. The second contains the sequence numbers of the STMT.ENTRYs raised and the third the sequence numbers of any CATEG.ENTRYs. The examples on the following pages illustrate the use of this field more clearly. Example 1: When a payment record is authorised a debit entry for the payment amount will be posted to the account entered PAYMENT.ACCOUNT field. The system will generate a statement entry and record the ID of the entry in this field as follows: 22.1 STMT.NO ............ 673732684.48 22.2 STMT.NO ............ 1 The number automatically generated in Field 22.1 is unique to the payment in question and is built up in the following way: 1. The first four digits '6737' indicate the number of days since the 1st of January 1968. In our example it corresponds to the 11th of June 1986. 2. The next five digits '32684' indicate the number of seconds since midnight. In our example it corresponds to an entry generated on line at 09:04:44 AM. 3. The last two digits '48' indicate the number of hundreds of seconds. The value assigned to Field 22.2 identifies the number in 22.1 as a statement entry number. This entry can be seen via the STMT.ENTRY Application using the the ID: 673732684.480001 where the suffix '0001' identifies it as the first statement entry (there may be more than one as the next example will illustrate). Example 2: A second example might be that of a payment where two separate charges have been levied. The payment itself will generate a debit statement entry as in the previous example. However, the addition of charges will generate a further two statement entries debiting the charge account for the amount of each charge, as well as a category entry crediting the relevant P&L category code(s). The IDs of the entries generated will be recorded as follows: 22.1 STMT.NO ............ 673732684.48 22.2 STMT.NO ............ 1 - 3 22.3 STMT.NO ............ 1 - 2 Once again the standard date-time stamp is generated and inserted in Field 22.1. Now, however, both fields 22.2 and 22.3 contain a value. The value in field 22.2 now indicates that three statement entries have been raised and in field 22.3 that two category entries have also been raised. To view these entries the STMT.ENTRY and now the CATEG.ENTRY application should be used. In this case the IDs required for each application are: STMT.ENTRY 673732684.480001 673732684.480002 673732684.480003 CATEG.ENTRY 673732684.480001 673732684.480001 Validation Rules: Internal multivalue field.

        public List<TField> getStmtNo()
        Returns the collection of StmtNo

        Field description: A number generated automatically by the system which identifies either a statement entry or a category entry in respect of a contract. Whenever a payment is entered via this application a number of accounting entries will be raised. This field will record the IDs of the entries generated. The IDs to accounting entries, both STMT.ENTRYs and CATEG.ENTRYs, consist of a date-time stamp and a sequence number. All entries raised by a particular input/transaction will have the same date-time stamp but different sequence numbers. This field has three multivalues, the first contains the base date-time stamp which is common to all entries. The second contains the sequence numbers of the STMT.ENTRYs raised and the third the sequence numbers of any CATEG.ENTRYs. The examples on the following pages illustrate the use of this field more clearly. Example 1: When a payment record is authorised a debit entry for the payment amount will be posted to the account entered PAYMENT.ACCOUNT field. The system will generate a statement entry and record the ID of the entry in this field as follows: 22.1 STMT.NO ............ 673732684.48 22.2 STMT.NO ............ 1 The number automatically generated in Field 22.1 is unique to the payment in question and is built up in the following way: 1. The first four digits '6737' indicate the number of days since the 1st of January 1968. In our example it corresponds to the 11th of June 1986. 2. The next five digits '32684' indicate the number of seconds since midnight. In our example it corresponds to an entry generated on line at 09:04:44 AM. 3. The last two digits '48' indicate the number of hundreds of seconds. The value assigned to Field 22.2 identifies the number in 22.1 as a statement entry number. This entry can be seen via the STMT.ENTRY Application using the the ID: 673732684.480001 where the suffix '0001' identifies it as the first statement entry (there may be more than one as the next example will illustrate). Example 2: A second example might be that of a payment where two separate charges have been levied. The payment itself will generate a debit statement entry as in the previous example. However, the addition of charges will generate a further two statement entries debiting the charge account for the amount of each charge, as well as a category entry crediting the relevant P&L category code(s). The IDs of the entries generated will be recorded as follows: 22.1 STMT.NO ............ 673732684.48 22.2 STMT.NO ............ 1 - 3 22.3 STMT.NO ............ 1 - 2 Once again the standard date-time stamp is generated and inserted in Field 22.1. Now, however, both fields 22.2 and 22.3 contain a value. The value in field 22.2 now indicates that three statement entries have been raised and in field 22.3 that two category entries have also been raised. To view these entries the STMT.ENTRY and now the CATEG.ENTRY application should be used. In this case the IDs required for each application are: STMT.ENTRY 673732684.480001 673732684.480002 673732684.480003 CATEG.ENTRY 673732684.480001 673732684.480001 Validation Rules: Internal multivalue field.

        public void setStmtNo(TField param, int idx)
        Set the specified StmtNo to a specific position in collection.
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: A number generated automatically by the system which identifies either a statement entry or a category entry in respect of a contract. Whenever a payment is entered via this application a number of accounting entries will be raised. This field will record the IDs of the entries generated. The IDs to accounting entries, both STMT.ENTRYs and CATEG.ENTRYs, consist of a date-time stamp and a sequence number. All entries raised by a particular input/transaction will have the same date-time stamp but different sequence numbers. This field has three multivalues, the first contains the base date-time stamp which is common to all entries. The second contains the sequence numbers of the STMT.ENTRYs raised and the third the sequence numbers of any CATEG.ENTRYs. The examples on the following pages illustrate the use of this field more clearly. Example 1: When a payment record is authorised a debit entry for the payment amount will be posted to the account entered PAYMENT.ACCOUNT field. The system will generate a statement entry and record the ID of the entry in this field as follows: 22.1 STMT.NO ............ 673732684.48 22.2 STMT.NO ............ 1 The number automatically generated in Field 22.1 is unique to the payment in question and is built up in the following way: 1. The first four digits '6737' indicate the number of days since the 1st of January 1968. In our example it corresponds to the 11th of June 1986. 2. The next five digits '32684' indicate the number of seconds since midnight. In our example it corresponds to an entry generated on line at 09:04:44 AM. 3. The last two digits '48' indicate the number of hundreds of seconds. The value assigned to Field 22.2 identifies the number in 22.1 as a statement entry number. This entry can be seen via the STMT.ENTRY Application using the the ID: 673732684.480001 where the suffix '0001' identifies it as the first statement entry (there may be more than one as the next example will illustrate). Example 2: A second example might be that of a payment where two separate charges have been levied. The payment itself will generate a debit statement entry as in the previous example. However, the addition of charges will generate a further two statement entries debiting the charge account for the amount of each charge, as well as a category entry crediting the relevant P&L category code(s). The IDs of the entries generated will be recorded as follows: 22.1 STMT.NO ............ 673732684.48 22.2 STMT.NO ............ 1 - 3 22.3 STMT.NO ............ 1 - 2 Once again the standard date-time stamp is generated and inserted in Field 22.1. Now, however, both fields 22.2 and 22.3 contain a value. The value in field 22.2 now indicates that three statement entries have been raised and in field 22.3 that two category entries have also been raised. To view these entries the STMT.ENTRY and now the CATEG.ENTRY application should be used. In this case the IDs required for each application are: STMT.ENTRY 673732684.480001 673732684.480002 673732684.480003 CATEG.ENTRY 673732684.480001 673732684.480001 Validation Rules: Internal multivalue field.

        public void setTransactionType(TField value)
        Set the specified value for TransactionType
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Defines the type of transaction being entered. The value entered in this field determines the type of transaction being processed. This will determine the transaction codes used and therefore the narrative appearing on the customers's statement. The following types are available: 1. REDEMPTION This type should be used for the input of partial redemption transactions, where an amount of principal less than the outstanding balance of the loan is being repaid. The value date for this type of transaction can be any date prior to the current date and may be prior to a previous repayment date if required. The transaction code used for this type is 810. Any adjustments to existing payments caused by this transaction being backdated prior to a previous repayment, will raise entries using the transaction codes 814 and 816 for principal and interest respectively. 2. CLOSURE This type of transaction can only be used to close or fully repay a contract. It is not necessary to enter the PAYMENT.AMOUNT as this will automatically default to the outstanding principal on the contract. Interest will be accrued up to the payment date in the VALUE.DATE field and this and any other outstanding interest will also be repaid, however this amount is not included in that shown in the PAYMENT.AMOUNT field. The transaction codes used for the accounting entries raised are 810 and 804 for principal and interest respectively. CLOSURE type transactions cannot be reversed. 3. ADJUSTMENT This type is similar to REDEMPTION except that the transaction code used for the debit entry of the PAYMENT.AMOUNT is 814. Validation Rules: 1 - 12 alpha character. Must be either (R)EDEMPTION, (C)LOSURE or (A)DJUSTMENT. (mandatory input) Must be CLOSURE if the PAYMENT.AMOUNT field contains the outstanding principal balance of the contract. CLOSURE only allowed if the amount entered in the PAYMENT.AMOUNT field is the outstanding principal balance of the contract.
