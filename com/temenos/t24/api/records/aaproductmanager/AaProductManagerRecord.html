      • getAction

        public TField getAction()
        Returns the Action

        Field description: The action that is being performed on this product. When a product has been completely designer, it undergoes a process called Publishing which comprises of actions at various levels. The following are the various options:

        1. PROOF - The product has been designed and reacy to be proofed. This is the first step in any publishing process and performs the following tasks: Expands the product definition to include the inherited property definitions from the published parent product Ensures that the resolved product definition includes all mandatory properties identified in the Product Group definition. Ensures that conditions exist for each property condition linked to a product. Ensures that a currency condition exists for each allowed currency for the product. Cross-validates product property conditions to ensure consistency and avoid conflicting values

        Product proofing will use the current Product Designer definitions for the product to perform the proofing validation. The proofing process will return any errors found in the validation process (updated in PRODUCT.ERRORS field) and these must be fixed before the product can be published.

        2. CANCEL.PROOF :

        After a successful/partially sucessfull proffing process, if the user wishes to cancel the proofing process i.e. delete the PRF records which may have been updated, he may choose this option which would bring the product back to the stage before proofing process was attempted.

        3. PUBLISH :

        After the product has been successfully proofed, then the next step is to update this product to catalog and make it available for sale to the customers. The process of moving files from PROOF to CATALOG is achieved by this option. When there are proofing errors, the product cannot be published. Also the status of the product in AA.PRODUCT file should be 'PROOFED' to attempt this option. The rules which govern the proofing process(with respect to hierarchy) apply to publishing as well.

        4. PENDING.PROOF.PUBLISH : Action to proof the product through T24.AUTHORISE service during update mechanism. 5. WITHDRAW : For future use 6. CLOSE : For future use

        Returns:

      • getProductVersion

        public TField getProductVersion()
        Returns the ProductVersion

        Field description: The version that this product is running on. This field is updated by the system and would be incremented by 0.1 for every action that the user performs on the product. It starts with 0.0 and the first proofing process would update to 0.1. Subsequent proofing/Cancel proofing action would increment it by 0.1. When the product is successfully published, it moves to the next whole number. In this case, when the product is published successfully, the version would move to 1.0. Further processing on this product would update version to 1.1, 1.2 etc.

        Returns:
        TField - The ProductVersion

      • setAction

        public void setAction(TField value)
        Set the specified value for Action
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: The action that is being performed on this product. When a product has been completely designer, it undergoes a process called Publishing which comprises of actions at various levels. The following are the various options:

        1. PROOF - The product has been designed and reacy to be proofed. This is the first step in any publishing process and performs the following tasks: Expands the product definition to include the inherited property definitions from the published parent product Ensures that the resolved product definition includes all mandatory properties identified in the Product Group definition. Ensures that conditions exist for each property condition linked to a product. Ensures that a currency condition exists for each allowed currency for the product. Cross-validates product property conditions to ensure consistency and avoid conflicting values

        Product proofing will use the current Product Designer definitions for the product to perform the proofing validation. The proofing process will return any errors found in the validation process (updated in PRODUCT.ERRORS field) and these must be fixed before the product can be published.

        2. CANCEL.PROOF :

        After a successful/partially sucessfull proffing process, if the user wishes to cancel the proofing process i.e. delete the PRF records which may have been updated, he may choose this option which would bring the product back to the stage before proofing process was attempted.

        3. PUBLISH :

        After the product has been successfully proofed, then the next step is to update this product to catalog and make it available for sale to the customers. The process of moving files from PROOF to CATALOG is achieved by this option. When there are proofing errors, the product cannot be published. Also the status of the product in AA.PRODUCT file should be 'PROOFED' to attempt this option. The rules which govern the proofing process(with respect to hierarchy) apply to publishing as well.

        4. PENDING.PROOF.PUBLISH : Action to proof the product through T24.AUTHORISE service during update mechanism. 5. WITHDRAW : For future use 6. CLOSE : For future use

        Parameters:
        value - The value to set
        Returns:
        nothing

        public void setProductVersion(TField value)
        Set the specified value for ProductVersion
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: The version that this product is running on. This field is updated by the system and would be incremented by 0.1 for every action that the user performs on the product. It starts with 0.0 and the first proofing process would update to 0.1. Subsequent proofing/Cancel proofing action would increment it by 0.1. When the product is successfully published, it moves to the next whole number. In this case, when the product is published successfully, the version would move to 1.0. Further processing on this product would update version to 1.1, 1.2 etc.
