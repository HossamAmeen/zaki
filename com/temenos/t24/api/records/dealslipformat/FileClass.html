      • getFile

        public TField getFile()
        Returns the File

        Field description: The name of the file from which data will be extracted. The first file, entered in the multi-value set, will become the 'default' file when specifying the fields to be output, hence, this should be the application name. Any related file can be defined, in subsequent multi-values. The key to the related file must be a field on the main contract (or a field on another related file). All the records from the related FILEs will be read in when the deal slip is being produced. The main contract record is taken directly from the screen. Validation Rules: 1-24 any characters (1) Must be a valid entry on the FILE.CONTROL file and have a valid STANDARD.SELECTION record.

        Returns:

      • getId

        public TField getId()
        Returns the Id

        Field description: The field name which holds the key to the FILE. The ID to the related file should be a field on the main contract. E.g. To access the customer record within the FOREX application then this should be entered as FOREX>COUNTERPARTY. ID can be entered in the following formats: -1>4 File 1, Field 4 5 File 1, Field 5 ACCOUNT>8 Account file field 8 ACCOUNT>CURRENCY Account file currency field "text" Constant eq "SYSTEM" The file number is the multivalue number of the FILE definition. If file or field numbers are entered they are immediately translated to their correct names. If the file number is 1, the default, then just the field name is returned. Eg. Producing a deal slip from the FOREX application requires data from FOREX itself, CUSTOMER, CURRENCY and the DEPT.ACCT.OFFICER files. The FILE & ID specifications would look like this: FILE ID 3.1 FOREX 4.1 @ID 3.2 CUSTOMER 4.2 FOREX>COUNTERPARTY 3.3 CURRENCY 4.3 FOREX>CURRENCY.BOUGHT 3.4 DEPT.ACCT.OFFICER 4.4 CUSTOMER>ACCOUNT.OFFICER Note : The id to the DEPT.ACCT.OFFICER record is a field on the CUSTOMER file which is in turn read using the COUNTERPARTY field of the FOREX contract. When defining this type of linkage make sure you enter the FILEs in their logical read sequence. When defining the ID for the application always use @ID. This will not actually be read when producing the deal slip but is instead taken from the details present on the screen. FIELD5.1 FIELD The name of the field to be printed. The name of the field (and from which file) or free format text can be entered. The format of the field specification is: 1>4 File 1, Field 4 5 File 1, Field 5 ACCOUNT>8 Account file field 8 ACCOUNT>CURRENCY Account file currency field "text" Constant eq "SYSTEM" !TOTAL.1 Total number 1 to be printed "SYSTEM.DATE" The current computer date Where file number is the multivalue number of the FILE definition. If file or field numbers are entered they are immediately translated to their correct names. If the file number is 1, the default, then just the field name is returned. A total can be printed instead of a field - see FORMAT for defining totals. When entering a field from the application then you can specify that the enrichment is printed rather than the field itself. This is entered as COUNTERPARTY ENRICHMENT or 4 E In this case the 4 E will be immediately translated to COUNTERPARTY ENRICHMENT. Validation Rules: 1-32 any characters FILE> FIELD or FIELD or "text" 1-40 Any Characters FILE> FIELD, FIELD or "text" or !TOTAL.nn
