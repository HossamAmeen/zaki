      • addPropertyVariation

        public void addPropertyVariation(java.lang.CharSequence param)
        Appends the specified PropertyVariation to the end of the collection.
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: The field is a sub-valued field with PROPERTY. This can only be defined for Properties of a Property Class for which variations are allowed. The user specifies in this field one or more variations that must exist for the related Property. If for example a product has both Platinum and Gold variations, it is possible that for one property (e.g. Principal Interest) the user wants to provide both Platinum and Gold benefits, however for another property (e.g. Penalty Interest) only the Platinum variation is specified and provides benefits. So, by using this field, user could state specific variation that this property would support and system would not insist on other variations stated in the VARIATION list. However a default condition must exist for all of these property conditions for the system to manoeuvre to when the specific variation condition is not met. Validation rules: 1. Allowed only for properties that support VARIATION. 2. If stated, should exist in the VARIATION list stated for this product. 3. Should be a valid entry in EB.LOOKUP of AA.PRODUCT.VARIATION 4. A default condition must exist for property conditions when this field is used.

        Parameters:
        param - the PropertyVariation to append
        Returns:
        nothing

      • getEffectiveBase

        public TField getEffectiveBase()
        Returns the EffectiveBase

        Field description: This field represent the effective base of the new property condition. It will accept two values. 1. START 2. ANNIVERSARY START – Forward condition effective date is calculated by considering base date as value date of arrangement. In case disbursement has happens different date than new arrangement creation date then all existing START type conditions will be deleted and new conditions created based on disbursement date of the arrangement ANNIVERSARY – Forward condition effective date is calculated by considering base date as anniversary date which is exist in account property field ANNIVERSARY. In case anniversary date has changed in life of the loan then all anniversary conditions will be deleted and new conditions created based on new anniversary date. ARRANGEMENT – It is default value of this field. If user doesn’t select any one of above options then system will treat those are ARRANGEMENT type conditions. Forward condition effective date is calculated by considering START.DATE of the arrangement. It will not be changed any of the time.

        Parameters:
        Returns:
        TField - The EffectiveBase
