        public TField getDrivingField()
        Returns the DrivingField

        Field description: Specifies the field in the driving file from which the list of keys to the main file will be taken. If driving field is blank all fields in all records found on the driving file are used as keys to the main file. If driving field is zero all keys on the driving file are used as keys to the main file. If driving field contains a field number the contents of this field in every record on the driving file is used as a key to the main file. Validation Rules: 1-3 numeric characters. May not be entered without a driving file specified.

      • getDrivingFile

        public TField getDrivingFile()
        Returns the DrivingFile

        Field description: Specifies another file from which a list of keys to the main file will be extracted. Normally the archiving system selects all records on the main file and considers them each in turn for archiving against the conditions defined in the parameter file record. If a driving file is used this allows only certain records from the main file to be selected and avoids the processing overhead involved in selecting very large files when it is known that only certain records will be suitable for archiving. Validation Rules: 1-27 alphanumeric characters. The filename entered must exist as an application on the PGM.FILE.

      • getHistoryType

        public TField getHistoryType()
        Returns the HistoryType

        Field description: Specifies the type of archiving treatment to be applied to the file. TYPE 1 - HISTORY CONTRACT Files with history. No action is taken on records which still exist in the live file. Records which exist entirely in history will be archived providing all conditions are satisfied. TYPE 2 - HISTORY NON-CONTRACT Files with history. Both the live record and its history records are considered and will be archived providing all conditions are satisfied. TYPE 3 - LIVE Live files only. Records will be archived from the live file if all conditions are satisfied. TYPE 4 - LIVE + HISTORY Files with history. The live record is first moved to history and then the contents of history considered for archiving. The live record is not moved unless all conditions are first satisfied. TYPE 5 - SINGLE FIELD Contents of the field are removed as specified providing all conditions are satisfied. TYPE 6 - LIVE TO HISTORY Files with history. If all conditions are satisfied the live record is moved to the history file. Validation Rules: 1 numeric character between 1 and 6. Must be one of the following: 1. HISTORY CONTRACT 2. HISTORY NON-CONTRACT 3. LIVE 4. LIVE + HISTORY 5. SINGLE FIELD 6. LIVE.TO.HISTORY History types 1,2,4,6 invalid if application type is not "H" in the PGM.FILE. For history type 3 (LIVE) application type must be either "U", "L", or "T". History type 5 (SINGLE FIELD) requires application type of "H","U","L", or "T".

        Parameters:
        Returns:
        TField - The HistoryType

      • getWaitingMonths

        public TField getWaitingMonths()
        Returns the WaitingMonths

        Field description: Specifies the required age of history records before they may be considered for archiving. Decision is based on the date/time of authorisation of the record being considered and the record which replaced it. For example a record authorised on 12/03/88 was valid on the system until the next change made and authorised on 17/06/88. In this case specifying a waiting months of 3 would not cause the record from 12/03/88 to be archived until 3 months following the 17/06/88. Validation Rules: 1-2 numeric characters. Invalid for history types 3,5,6. Mandatory for history types 1 and 2.

        Returns:
        TField - The WaitingMonths

        public void setHistoryType(TField value)
        Set the specified value for HistoryType
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Specifies the type of archiving treatment to be applied to the file. TYPE 1 - HISTORY CONTRACT Files with history. No action is taken on records which still exist in the live file. Records which exist entirely in history will be archived providing all conditions are satisfied. TYPE 2 - HISTORY NON-CONTRACT Files with history. Both the live record and its history records are considered and will be archived providing all conditions are satisfied. TYPE 3 - LIVE Live files only. Records will be archived from the live file if all conditions are satisfied. TYPE 4 - LIVE + HISTORY Files with history. The live record is first moved to history and then the contents of history considered for archiving. The live record is not moved unless all conditions are first satisfied. TYPE 5 - SINGLE FIELD Contents of the field are removed as specified providing all conditions are satisfied. TYPE 6 - LIVE TO HISTORY Files with history. If all conditions are satisfied the live record is moved to the history file. Validation Rules: 1 numeric character between 1 and 6. Must be one of the following: 1. HISTORY CONTRACT 2. HISTORY NON-CONTRACT 3. LIVE 4. LIVE + HISTORY 5. SINGLE FIELD 6. LIVE.TO.HISTORY History types 1,2,4,6 invalid if application type is not "H" in the PGM.FILE. For history type 3 (LIVE) application type must be either "U", "L", or "T". History type 5 (SINGLE FIELD) requires application type of "H","U","L", or "T".

      • setWaitingMonths

        public void setWaitingMonths(java.lang.CharSequence value)
        Set the specified value for WaitingMonths
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Specifies the required age of history records before they may be considered for archiving. Decision is based on the date/time of authorisation of the record being considered and the record which replaced it. For example a record authorised on 12/03/88 was valid on the system until the next change made and authorised on 17/06/88. In this case specifying a waiting months of 3 would not cause the record from 12/03/88 to be archived until 3 months following the 17/06/88. Validation Rules: 1-2 numeric characters. Invalid for history types 3,5,6. Mandatory for history types 1 and 2.
