      • getAutNewContent

        public TField getAutNewContent()
        Returns the AutNewContent

        Field description: Enables the User to define the new automatic default value. Must be used in conjunction with AUTOMATIC OLD CONTENT (Field 44) and AUTOMATIC FIELD NUMBER (Field 43). A routine (preceded by "@") or an EB.API record of type METHOD which implements an interface defined in the EB.API record VERSION.CHECK.REC.RTN.HOOK may be specified instead of actual values. The routine will only be executed if the EXC.INC.RTN field is set to Y. See the EB.API record VERSION.CHECK.REC.HOOK for the full list of supported interfaces, initially EB.TemplateHook.defaultFieldValues(). Input in this field must be in accordance with the field definition of the selected Application. Quotes should only be used where the new data actually requires quotes e.g if a new value is required to be YES it should not be input as "YES". It is important to clearly understand the relation between this field and AUTOMATIC OLD CONTENT. The following examples will clarify their use. 1. When the User wants to create an automatic default at the creation of new record ID's, he should then make the following input in these three fields : AUTOMATIC FIELD NUMBER - Indicates the field number for which you want to create an automatic default. AUTOMATIC OLD CONTENT - Must remain blank. AUTOMATIC NEW CONTENT - Enter the value which you want to default automatically. 2. When you want to replace automatically an old content by a new content, you should then make the following input in these three fields : AUTOMATIC FIELD NUMBER - Indicate the field number for which you want to change automatically the old content. AUTOMATIC OLD CONTENT - Enter the old content AUTOMATIC NEW CONTENT - Enter the new content which you want to replace automatically. 3. When you want to run a subroutine which will default a value in a particular field, you should make the following input in these three fields : AUTOMATIC FIELD NUMBER - Indicates the field number for which you want to create a automatic default AUTOMATIC OLD CONTENT - Must remain blank AUTOMATIC NEW CONTENT - Enter the subroutine name which needs to be executed for the the field in AUTOMATIC.FIELD.NUMBER. Note: If the field AUTOMATIC OLD CONTENT is specified, only records containing the specified value will be changed when using this Version. If the field AUTOMATIC OLD CONTENT is not specified, the content specified in the field AUTOMATIC NEW CONTENT (Field 44) will be generated when creating a new record with this Version. The value of the AUT.NEW.CONTENT field can be set using a system (e.g. TODAY, COMPANY, etc.) or user-defined (e.g. CURRENT.CUSTOMER, CURRENT.ACCOUNT, etc.) variable. In order to use the value of a variable an "!" character must be used preceding the variable name. For example, a field can be defaulted to today's date by using "!TODAY" or the value of a user defined variable containing a customer number by using "!CURRENT.CUSTOMER". Validation Rules: Up to 35 type A (alphanumeric) characters. Multivalue field associated with fields AUTOMATIC FIELD NUMBER and AUTOMATIC OLD CONTENT. (Optional input. No default value). Input in this field is not allowed when PRINT ONLY (Field 1) contains the value of Y. Duplication of field numbers is not allowed. Input in this field is not allowed when AUTOMATIC FIELD NUMBER has been left blank. Input of the Old and the New must be different. a) When a subroutine or an EB.API record of type METHOD which implements an interface needs to be executed then the subroutine name should preceed with an @ symbol b) The subroutine entered should exist in PGM.FILE with PGM.TYPE = 'S'. c) The VERSION ID should exist in field APPR.FOR.SUBR in the PGM.FILE record for the subroutine. d) The PGM.TYPE for the VERSION record should be one of the following :- 'U', 'H', 'W', 'M', 'S', 'B'

      • getAutOldContent

        public TField getAutOldContent()
        Returns the AutOldContent

        Field description: Enables the User to define the previous field content which is to be changed automatically with the new default value as held in the next field. Must be used in conjunction with AUTOMATIC NEW CONTENT (Field 45) and AUTOMATIC FIELD NUMBER (Field 43). This field can either be a constant e.g 100, YES, 21050 etc or a pattern match specification e.g 3N0X, 5A5N etc. To match any input use 0X. It is recommended that constants are enclosed in quotes. This field is only used when records containing a certain existing value should automatically be changed to contain the new value specified in the AUTOMATIC NEW CONTENT field. Validation Rules: Up to 35 type A (alphanumeric) characters. Multivalue field associated with Fields 43 to 45. (Optional input. No default value). Input in this field is not allowed when PRINT ONLY (Field 1) contains the value of Y. Input in this field is not allowed when AUTOMATIC FIELD NUMBER (Field 43) has been left blank. Input of the Old and the New Automatic Content must be different.

      • getAutomFieldNo

        public TField getAutomFieldNo()
        Returns the AutomFieldNo

        Field description: Enables the User to define the field name which is to be subject to an automatic default value. It must be used in conjunction with the fields Automatic Old/New Content. The definition of the value or sub-value numbers must always be included in this field. Field names can be entered in the format: a) Field name (e.g. MNEMONIC for field 1 in the customer file followed by, b) '-' when the User wants to define a multivalue field, followed by, c) Value Number (e.g. RELATION-1 for RELATION in the customer file) when the User wants to define multivalue fields, followed by, d) '.' when the User wants to define a subvalue field, followed by, e) Sub-Number when the User wants to define a subvalue field. Alternatively, field names can be entered as field numbers, as follows: a) Field number (e.g. 1 for MNEMONIC in the CUSTOMER file), followed by, b) '.' when the User wants to define a multivalue field, followed by, c) 'Value number' (e.g. 7.1 for RELATION in the CUSTOMER file) when the User wants to define multivalue fields, followed by, d) '.' when the User wants to define a subvalue field, followed by, e) 'Sub number' when the User wants to define a subvalue field. Field numbers will be translated to field names when entered. For local reference field use the format LOCAL.REF-nn. Validation Rules: Up to 40 type A (alphanumeric) characters in the format XXXXXXXXXX.NNN.NN or NNN.NNN.NN Multivalue field associated with Fields 43 to 45. (Optional input. No default value). Only field names defined in the corresponding Application are allowed. Input of field names relating to Internal or External fields (automatically generated by the System) is not allowed. Input in this field is not allowed when PRINT ONLY (Field 1) contains the value of Y. Duplication of fields names is not allowed.

        Returns:
