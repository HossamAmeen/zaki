      • addOutDeliveryRef

        public void addOutDeliveryRef(TField param)
        Appends the specified OutDeliveryRef to the end of the collection.
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Contains the outward delivery reference number of the Account statements(MT941/MT942/MT940/MT950) which is generated in response to this Inward MT920 request. Request received for Interim statement (941/942) is processed online & this field is updated with the relevant delivery reference. Account Statements (940/950) are generated as a special statement during End of day process and the delivery reference number is updated here & respective Account record in ACCOUNT.STATEMENT during end of day processing. Incase a statement is already scheduled for a Message type in frequency 1 (date as given in STMT.FQU.1 field in ACCOUNT.STATEMENT) and an Inward MT920 is processed for the same message type, then statement is generated as per frequency 1 only and delivery reference is updated in ACCOUNT.STATEMENT alone. This field is updated as "MSG ALREADY SCHEDULED TO BE SENT TODAY" when more than one MT920 request is received with same Message type-either (940/950). The Outward Message is mapped , formatted and generated using the Carrier & Address as defined for the combination of Customer / Account / Message type in the DE.PRODUCT Application as per the existing Delivery mechanism. For example, To send Swift Message MT941 for the Accounts related to Dell to Merrill Lynch Bank in response to MT920, following setup is required. Add a record in DE.ADDRESS for the Customer (1044) & Carrier (Swift ) combination as "MLNYUS33" (Swift address of Correspondent Bank- Merrill Lynch Bank) DE.PRODUCT should be set to carrier as 'SWIFT' for the Customer(1044) & Message (941) combination. If XML.STMT.TYPE is specified, then this field holds the XML.OUTPUT.MSG id XML.OUTPUT.MSG id is updated after the XML.TRANSFORMATION service Validation Rules: No input -Updated by system Multi value field Must be a valid record in DE.O.HEADER for MT messages Must be a valid record in XML.OUTPUT.MSG for CAMT messages

        Parameters:
        param - the OutDeliveryRef to append
        Returns:
        nothing

      • getInstitution

        public TField getInstitution()
        Returns the Institution

        Field description: Contains the Receiver Institution details of the MT920 (Request Message) for Outward instruction and the Sender Institution details for Inward instruction. Outward Specify a valid BIC prefixed with "SW-" or a Customer in this field and the outward MT920 is generated with the receiver as specified here along with the details as given in below fields. Inward For Inward MT920 which are processed through Inward processing, this field contains the Original Sender of MT920 ie. either a BIC address prefixed with "SW-" or a Customer number when the BIC has a record in DE.ADDRESS. To send the statement to the Sender of MT920 message, the DE.ADDRESS record for the Incoming Accounts' Customer should have the Correspondent BIC code otherwise statements are sent to the address and carrier as applicable for the Incoming Account's Customer instead of Correspondent. Validation Rules: Mandatory field For outward instruction - Must be a valid BIC prefixed with "SW-" or a valid T24 Customer Number. For inward instruction -updated either customer number or BIC prefixed with "SW-".

        Parameters:
        Returns:
        TField - The Institution

        Field description: Contains the outward delivery reference number of the Account statements(MT941/MT942/MT940/MT950) which is generated in response to this Inward MT920 request. Request received for Interim statement (941/942) is processed online & this field is updated with the relevant delivery reference. Account Statements (940/950) are generated as a special statement during End of day process and the delivery reference number is updated here & respective Account record in ACCOUNT.STATEMENT during end of day processing. Incase a statement is already scheduled for a Message type in frequency 1 (date as given in STMT.FQU.1 field in ACCOUNT.STATEMENT) and an Inward MT920 is processed for the same message type, then statement is generated as per frequency 1 only and delivery reference is updated in ACCOUNT.STATEMENT alone. This field is updated as "MSG ALREADY SCHEDULED TO BE SENT TODAY" when more than one MT920 request is received with same Message type-either (940/950). The Outward Message is mapped , formatted and generated using the Carrier & Address as defined for the combination of Customer / Account / Message type in the DE.PRODUCT Application as per the existing Delivery mechanism. For example, To send Swift Message MT941 for the Accounts related to Dell to Merrill Lynch Bank in response to MT920, following setup is required. Add a record in DE.ADDRESS for the Customer (1044) & Carrier (Swift ) combination as "MLNYUS33" (Swift address of Correspondent Bank- Merrill Lynch Bank) DE.PRODUCT should be set to carrier as 'SWIFT' for the Customer(1044) & Message (941) combination. If XML.STMT.TYPE is specified, then this field holds the XML.OUTPUT.MSG id XML.OUTPUT.MSG id is updated after the XML.TRANSFORMATION service Validation Rules: No input -Updated by system Multi value field Must be a valid record in DE.O.HEADER for MT messages Must be a valid record in XML.OUTPUT.MSG for CAMT messages
