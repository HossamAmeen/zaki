        public TField getRevolType()
        Returns the RevolType

        Field description: This field is used to indicate whether the Facility granted if REVOLVING or NON REVOLVING type. A revolving type facility is re-instated on repayment of the drawings made (a classic example being working capital finance), while a non-revolving facility is of reducing type and may be drawn only once. If LIQUIDATION.MODE chosen is INTERNAL, a Revolving Facility is re-instated only if the principal portion is repaid using SL.OD.SETTLE application. For option AUTOMATIC in field LIQUIDATION.MODE such reinstatement would happen on processing of maturity of the loan. Validation Rules: Only wither of two values REVOLVING or NON-REVOLVING is accepted Value input here would get defaulted to FACILITY. Associated multi-value field under SL.CURRENCY

      • getUwDevol

        public TField getUwDevol()
        Returns the UwDevol

        Field description: YES/NO indicator to show whether there has been any devolvement of underwriting. In the event that aggregate subscriptions received are less than the mandate amount, the difference is devolved amongst the underwriters in the agreed proportion. The apportionment of the un-subscribed portion amongst the underwriters are done manually as the manner of devolvement would differ on a case to case basis. This field indicates whether there has been any devolvement or not. Validation Rules: Optional input. Only allowed values are YES and NO. Value Null means NO.

        Parameters:
        Returns:
        TField - The UwDevol

      • getUwFeeAcc

        public TField getUwFeeAcc()
        Returns the UwFeeAcc

        Field description: Indicates the liquidation account for the underwriting fee. Account mentioned in this field should be denominated in the same currency of as in field SL.CURRENCY and this rule is applicable within each multi-value set. On creation of FACILITY, accounting entries would be raised debiting this account for the aggregate underwriting fee payable for the respective facility. In case the underlying Contract Customer (eg. Customer.id in MONEYMARKET,Counterparty in FOREX, etc.) has a record in CUSTOMER.SSI application, then the account number in this field gets defaulted as per the definition for that Customer from CUSTOMER.SSI.For more information on the usage of this feature please refer to CUSTOMER.SSI application. Validation Rules: Account input in this field should be denominated in the same currency as SL.CURRENCY. Must be a valid ID on the ACCOUNT table.
