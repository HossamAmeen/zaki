        public TField getApplication()
        Returns the Application

        Field description: Specifies the name of an existing Application to be included in EB.USER.ROLES record.

        E.g. the name of an existing application USER.SMS.GROUP preceded by '@' sign,

        ALL.PG which allows access to all Applications or

        ALL.PG.(program type) which allows access to all Applications for a certain type (see below). The following program types can be used in conjunction with ALL.PG (e.g. ALL.PG.H) to give access to a particular type of program:

        Program Type Meaning

        H Program - 3 standard files exist - Live, Unauthorised and History, e.g. CUSTOMER.

        U Program - Live and Unauthorised files exist, but no History, e.g. VERSION.

        L Program - only a Live file exists (updated automatically by the system) e.g. STMT.ENTRY (only L, S and P functions can be used with this type of program).

        M Program - designed to do other than processing the standard T24 functions using a specific file e.g. PASSWORD.ENABLE, End Of Day Batch programs.

        T Program - handles automatically updated CONCAT files, e.g. CUSTOMER.ACCOUNT (only L, S and P functions can be used with this type of program). Note: The following 3 type M programs are available to all users and cannot be restricted: SIGN.ON SIGN.OFF PASSWORD E.g. 1 To define a role which give a User access to all the records for the Company this role is defined for, using all Functions (except auditor review) and all Versions of all Applications. Enter the following:

        Field 2.1(APPLICATION) ALL.PG

        Field 3.1(VERSION)

        Field 4.1(FUNCTION) ALL Note: Fields 5-8 are used to specify particular records or types of records which may be accessed within an Application. See FIELD NO (Field 5) for examples. E.g. 2 To give a User access to List Display, Input and Copy Customer records and to List and Display records for all other Applications. Enter the following:

        Field 2.1(APPLICATION) ALL.PG

        Field 4.1(FUNCTION) L S

        Field 2.2(APPLICATION) CUSTOMER

        Field 4.2(FUNCTION) L S I C E.g. 3 To prevent a USER from using any type M Applications, but allow him to List and Display records in any other Applications. Enter the following:

        Field 2.1(APPLICATION) ALL.PG.H

        Field 4.1(FUNCTION) L S

        Field 2.2(APPLICATION) ALL.PG.U

        Field 4.2(FUNCTION) L S

        Field 2.3(APPLICATION) ALL.PG.L

        Field 4.3(FUNCTION) L S

        Field 2.4(APPLICATION) ALL.PG.T

        Field 4.4(FUNCTION) L S Validations:

        1-30 type SSS (uppercase alpha) characters, Application name.

        ALL.PG to allow all Applications.

        ALL.PG (program type) see details.

        @ sign followed by a valid USER.SMS.GROUP

        public TField getFieldNo()
        Returns the FieldNo

        Field description: Specifies (when used together with Fields 6.1 - 8.1) precisely, which record can be accessed, that are specified in the corresponding APPLICATION (Field 2.) A record can be accessed, only if the field indicated, has a value allowed in the corresponding Fields 6 - 8 0-3 numeric characters. Or Up to 55 characters (type ANY) formatted in the following way:

        Up to 30 characters (type ANY) must be a valid entry on the FILE.CONTROL file, followed by,

        A gt (greater than) sign, followed by,

        Up to 2 characters (type A) must be a valid entry in the multi-value field SYS.FIELD.NAME or USER.FIELD.NAME on the STANDARD.SELECTION record for the file specified in (i). E.g. FOREX COUNTERPARTY It is recommended that field names and not field numbers are specified. If field numbers are specified, then when a file layout changes all the user profiles affected will be manually changed. E.g. 1: To allow a User to See and List all Accounts records, but only Input (or change), Copy or Delete Accounts in pounds sterling.

        Field 2.1(APPLICATION) ACCOUNT

        Field 4.1(FUNCTION) L S

        Field 2.2(APPLICATION) ACCOUNT

        Field 4.2(FUNCTION) I C D

        Field 5.2(FIELD NO.) ACCOUNT>CURRENCY (currency)

        Field 6.2(DATA COMPARISON) EQ (equal)

        Field 7.2(DATA FROM) GBP E.g. 2: To allow a User to access Account records, only if they have Currency equal GBP and Category not in the range 01-001 to 01-099.

        Field 2.1(APPLICATION) ACCOUNT

        Field 4.1(FUNCTION) ALL

        Field 5.1(FIELD.NO) ACCOUNT>CURRENCY (Currency)

        Field 6.1(DATA.COMPARISON) EQ (equal)

        Field 7.1(DATA.FROM) GBP

        Field 2.2(APPLICATION) ACCOUNT

        Field 4.2(FUNCTION) ALL

        Field 5.2(FIELD.NO) ACCOUNT>CATEGORY (Category)

        Field 6.2(DATA.COMPARISON) NE (not equal)

        Field 7.2(DATA.FROM) 01-001

        Field 8.2(DATA.TO) 01-099 E.g. 3: To allow a User to Authorise Data Capture records, but only if they do not contain an override.

        Field 2.1(APPLICATION) DATA.CAPTURE

        Field 4.1(FUNCTION) A

        Field 5.1(FIELD.NO) DATA.CAPTURE>OVERRIDE (Override)

        Field 6.1(DATA COMPARISON) LT (Less Than)

        Field 7.1(DATA.FROM) (lowest possible value) Note: It is not possible to prevent a user from entering on override during input, but Authorization can be restricted, as above, to ensure that any overrides are checked by a more senior person is the lowest value a field can have, so specifying LT (Less Than) ensures that no override may be present. E.g. 4: Restricting FOREX transaction to counterparties, that belong to a specific account officer.

        Field 2.1(APPLICATION) FOREX

        Field 4.1(FUNCTION) I

        Field 5.1(FIELD.NO) CUSTOMER>ACCOUNT.OFFICER

        Field 6.1(DATA COMPARISON) EQ

        Field 7.1(DATA.FROM) 100 Note: The field specified does not have to belong to the application. If the system can link the APPLICATION to the field, then the SMS restriction will be applied. In this example the COUNTERPARTY field on the FOREX file provides the link to the CUSTOMER file. Hence when the transaction is input, T24 reads the CUSTOMER file using the COUNTERPARTY as the key and checks that the ACCOUNT.OFFICER is 100. You do not have to define COUNTERPARTY as the key to CUSTOMER file. It is done automatically based on the RELATED.FILE stored in the STANDARD.SELECTION record for FOREX. Note: If the user wants to test a value in a related file, a consderable process load can be incurred, when the user lists the file. This can be relieved by defining the related field as a J-descriptor user field on the STANDARD.SELECTION record. Thus in the example above, define CUST.OFFICER as type J, CUSTOMER,ACCOUNT.OFFICER and then specify the FIELD.NO as FOREXCUST.OFFICER. Validations:

        See Details for format.

        It must not be higher than the number of fields specified in APPLICATION (Field 2.)

        Input in this field will not be allowed when N has been input in FUNCTION (Field 4.1).

        Multi-valued field associated with Fields 2-4 and 6-8.

        (Optional input. No input means all records may be accessed, regardless of contents.)

        For multi-valued fields, only the field number may be specified, not the entry number, e.g. 5, not 5.1, all entries in the multi-value field will be checked. The system displays the name of the field indicated.

        public TField getFunction()
        Returns the Function

        Field description: Specifies the functions, which are used to access the records of the Application, entered in the corresponding Field 2. This field is used, together with the associated Fields 2-3 and 5-8 to specify precisely, which function, the user uses to access which record and in which Application. Valid function codes are: A 2 C D E H I L P Q R S V If NOT is entered, plus one or more Function codes, the system converts the code into a list of all the Functions which are allowed. (See Fields 2, 3 and 5 for examples.) Note:During an input function, even if the function C (copy) is not allowed, the user can transfer the ID (together with its enrichment) of a selected record using the ! facility. Validations:

        0-29 type S (uppercase alpha or numeric) characters.

        Function codes separated by spaces.

        NOT + one or more Function codes.

        N (No Functions allowed).

        ALL (All Functions allowed).

        Multi-valued field associated with Fields 2-3, 5-8.

        Mandatory input except if a USER.SMS.GROUP is defined in field 2.1, the input will not be allowed.

        No Default Value.

        If Function 2 (2nd Authorization) is allowed, but not A (1st Authorization), the User is allowed to enter Function Code A - a Security Violation is generated if the record requested is not ready for 2nd Authorization.

      • getUseLocalSms

        public TField getUseLocalSms()
        Returns the UseLocalSms

        Field description: Specifies whether the system should use a local API to define additional SMS rules This is a Yes/No field If set to Yes, the system will initially perform any validation rules defined in the applications USER, USER.SMS.GROUP and EB.USER.ROLES and will then use the fixed-named API routine EB.LOCAL.SMS.ROUTINE to provide any additional security checks The API signature is EB.LOCAL.SMS.ROUTINE(smsRestrict, smsType, smsApplication, smsId, smsRecord, smsField, smsValue) where smsRestrict should be returned as 1 if the data is to be restricted or 0 if not smsType is passed with the value ENQUIRY, RECORD or FIELD depending on the type of SMS smsApplication is passed with the application to be checked smsId is passed with the ID of the record to be checked smsRecord is passed with the contents of the record to be checked smsField is passed with the name of the field to be checked smsValue is passed with the contents of the field to be checked. If fieldValue contains more than 1 @FM delimited values, then it will be converted to the symbol (!) before passing to local API and the local SMS API will have to handle this accordingly For more information refer to the Security Management System section in the user guide.

        Returns:
        TField - The UseLocalSms

      • getVersion

        public TField getVersion()
        Returns the Version

        Field description: Specifies a Version, which is used to access the records, that are entered in the corresponding APPLICATION (Field 2.1). This field is used, together with the associated Fields 2-8, to specify precisely, is accessed by the user and in which Application , using which Versions and Functions.

        * and ? - specifies more than one Version. They act as wild card characters.

        * - replaces any number of characters including null, numbers, alphabets etc.

        ? - replaces one character including null, numbers, alphabets etc. The version field can have the values for e.g. ,SAVE* - which allows the user to access Versions of ACCOUNT applications for e.g. :ACCOUNT,SAVESTUDENT, ACCOUNT,SAVERESIDENT, ACCOUNT,SAVESALARIED, etc The version field can have the values for e.g.: ,SAVE1000?00- which allows the user to access versions of ACCOUNT application for e.g. SAVE1000100, SAVE1000200, SAVE1000300 , etc Note: Versions of Application, changes the screen layout or the rules for Input, Validation and Authorization. E.g. 1 To allow a User to List and See all Accounts except for those with Account Officer = 24 using the standard version of Account, and to List and See all Accounts for Account Officer = 24 only, using a special Version Account,99:

        Field 2.1(APPLICATION) ACCOUNT

        Field 3.1(VERSION)

        Field 0 4.1(FUNCTION)

        Field L S 5.1(FIELD NO)

        Field 9 6.1(DATA COMPARISON) NE

        Field 7.1(DATA FROM) 24

        Field 2.2(APPLICATION) ACCOUNT

        Field 3.2(VERSION)

        Field 99 4.2(FUNCTION) L S

        Field 5.2(FIELD NO) 9

        Field 6.2(DATA COMPARISON) EQ

        Field 7.2(DATA FROM) 24 E.g. 2 To allow a User to Input and See all Account version, with name starting with SAVINGS followed by any character:

        Field 2.1(APPLICATION) ACCOUNT

        Field 3.1(VERSION) ,SAVINGS*

        Field 4.1(FUNCTION) I S For more examples of Fields 2-8 see Fields 2 and 5. Validations:

        0-50 type A (alphanumeric) characters beginning with , or

        0 to allow only the standard version. or

        No input to allow all Versions.

        Multi-valued field associated with Fields 2, 2 and 4-8.

        Optional input, except if a USER.SMS.GROUP is defined in field 2.1, in which case the input is not allowed. If not entered, access to all Versions is allowed.

        Special character * will replace any number of characters and ? replaces one character.

        Also ? after * and two occurrences of * are not allowed.

        Parameters:
        Returns:
        TField - The Version
