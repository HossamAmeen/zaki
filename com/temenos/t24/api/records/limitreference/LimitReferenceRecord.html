      • addGetSubAlloc

        public void addGetSubAlloc(TField param)
        Appends the specified GetSubAlloc to the end of the collection.
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Specifies which Limit References may donate Suballocations to the Limit Reference being input. This field allows control over the Risk involved with Suballocating from a Limit Reference of one type to a Limit Reference of another type. Suppose that there are three Limit References:- 300 Overdraft 400 L/C s 500 Loans A given installation may decide that the Risk involved in these Limit References runs from Loans (highest) to L/C's (lowest). So the system can allow the allocation of some high-risk to a lower risk but not the other way around. The table below shows an example scenario.

        LIMIT.REFERENCE GET.SUB.ALLOC
        300 Overdrafts500
        400 L/C's300 and 500
        500 Loansnone
        Here you can allocate some of the Loans LIMIT to either LC's or overdrafts; some overdraft LIMIT to LC's; but you cannot allocate the lowest LIMIT of LC's to Loans or overdrafts. An override is generated in LIMIT.SUB.ALLOC when a sub-allocation is made from one reference to another that is not listed here. Validation Rules: 3 to 7 numeric or 3 to 12 Alpha Numeric character Limit Reference Code. Multivalue field. (Optional input; Default Value: None) Duplicate values of the Limit Reference code are not allowed. Input in this field must be a valid LIMIT.REFERENCE.

        Parameters:
        param - the GetSubAlloc to append
        Returns:
        nothing

        public void setSubProductLevel(TField value)
        Set the specified value for SubProductLevel
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: The level of the sub-product being defined. Up to 9 sub product levels can be defined below the standard sub product level. A standard sub-product can be identified by the last two characters of the limit reference being anything other than zeros. Sub-product levels 1 to 9 are allowed. With 1 representing the highest and 9 the lowest sub product levels.

        For Alpha Numeric Limit References User can define any Number of Sub Levels.

        There will be no restriction of up to how many levels can be set up, but Sub Product Level should not be defined.

        When Sub Product Levels are defined, Only the existing structure of 9 levels is allowed

        As long as a product holds a Reference Child, it means the respective product is a parent for the given child products

        Example Product & Sub-Product: We need a Limit structure to cover generic Loans that is sub-divided into those to Banks and those to Non-Banks. This requires a Product and two sub-products. We define: 5000 - product for Loans 5010 - sub-product for Loans to Banks 5020 - sub-product for Loans to Non-Banks Example Product, Sub-Product & Sub-product levels: We need a Limit structure to cover generic Loans that is sub-divided into those to Banks and those to Non-Banks. Each of these needs to be split into secured & non-secured. The secured level also needs to be split between secured by property; secured by cash & secured other. This requires a Product; two sub-products, two sub-products at level 1 and six sub-products at level 2. We define: 6000 - product for Loans 6010 - sub-product for Loans to Banks 6011 - secured (level 1) 6012 - secured by property (level 2) 6013 - secured by cash (level 2) 6014 - secured other (level 2) 6020 - sub-product for Loans to Non-Banks 6011 - secured (level 1) 6012 - secured by property (level 2) 6013 - secured by cash (level 2) 6014 - secured other (level 2) Visually the structure would be like: (SEE FIGURE BELOW) As the levels can go down to level 9, a very complex structure could be defined. Note: It is possible that the existing limit reference structure is such that it is not possible to define the required levels of sub division e.g.: - 400 defines loans at product level; 410 defines customer loans at sub-product level; 415 defines mortgages at sub-product level. There are not enough numbers available between 410 and 415 to define the new sub product levels. In this case a new structure should be defined to replace the existing customer loans structure and LIMIT and LIMIT.PARAMETER updated accordingly. New loans will then use the new limit structure, and changing the limit reference on existing loans will have the same effect. Validation Rules: Can only be defined for sub product level (last two characters of reference must be non zero) Must be defined in ascending order for new sub product levels. The sub product level can only be changed if it is not part of an existing limit structure i.e. the PARENT.REFERENCE and REFERENCE.CHILD fields are blank. Not allowed for Limit References in a Cross Limit structure i.e. when the field CROSS.LIMIT is set to one of TOP, CROSS or PROD.
