      • getIntSelPriority

        public TField getIntSelPriority()
        Returns the IntSelPriority

        Field description: Internal selection priority. This is a system generated field containing the full, internal selection priority for this record. When a database is specified as having unique.selection then each item of T24 data will be added into one and only one line, however it is possible that an item of data will match the selection criteria for more than one line. For example we have two lines - Current Accounts and Current Accounts for Banks. A current account for a bank could be placed in either line - it is both a current account and a banks current account. Obviously we want to choose the latter but how do we ensure that MI understands our intentions? MI offers a mechanism called Selection Priority. Firstly there is a field on the line description (mi.line) where we may input the lines priority. In the case of the above example, we could give the line Current Accounts for Banks a higher priority than Current Accounts. MI would then check the lines in priority sequence. Finding a match against Current Accounts for Banks it would allocate the item to this line and look no further. We have achieved a correct result. However allocating a manual selection priority to all lines would prove tedious. Consequently MI offers an automatic facility to default the selection priority based upon an analysis of how specific the definition of the selections are. Again in the above example, Current Accounts for Banks is a more specific selection than just Current Accounts and records would be compared to this first. The same result would have been achieved as when the priorities were allocated manually. MIs determines how specific a selection is by analysing the number of different operands and fields used in the selection. This process is described in more detail below. MI holds the sequence in which it will check lines in the field int.sel.priority on mi.line. This field consists of two parts: the selection.priority as input by the user and an analysis of the relative specificity of the selection parameters for the line. MI determines the relative priority of lines by analysing the mi.selection records that the mi.line record refers to. It examines three factors: - The number of different source fields used in the selection records - The number of different operands used on those fields - The relative priority of those operands (see the table below) For example if MI is presented with two lines with the following selections: Line 1 - PRODCAT EQ 1000 Line 2 - PRODCAT EQ 1000 AND CURRENCY EQ USD it will assign data to line 2 if it matches. Only if it does not match here will it check against line 1. MI holds its assessment of the relative priority of each line in the field internal selection priority on the mi.line record. Validation Rules: No input. System generated field.

      • getSelectionPriority

        public TField getSelectionPriority()
        Returns the SelectionPriority

        Field description: The priority sequence for this line record. 9 = highest priority. Compare to this record first. 1 = lowest priority. Compare to this record last. Indicates to MI the sequence in line records are to be compared to source records to see of ths source is to be selected for this line. Input to this field should not usually be required because the MI application will make its own estimate based upon the complexity of the selection records. See the helptext for INT.SEL.PRIORITY for further details. Only allowed if values are to be allocated to only one line in the report (i.e. if unique.selection on the report type record is set to Y or blank). Validation Rules: Mandatory input. Numeric 1-9. Default is 5.

      • setIntSelPriority

        public void setIntSelPriority(java.lang.CharSequence value)
        Set the specified value for IntSelPriority
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Internal selection priority. This is a system generated field containing the full, internal selection priority for this record. When a database is specified as having unique.selection then each item of T24 data will be added into one and only one line, however it is possible that an item of data will match the selection criteria for more than one line. For example we have two lines - Current Accounts and Current Accounts for Banks. A current account for a bank could be placed in either line - it is both a current account and a banks current account. Obviously we want to choose the latter but how do we ensure that MI understands our intentions? MI offers a mechanism called Selection Priority. Firstly there is a field on the line description (mi.line) where we may input the lines priority. In the case of the above example, we could give the line Current Accounts for Banks a higher priority than Current Accounts. MI would then check the lines in priority sequence. Finding a match against Current Accounts for Banks it would allocate the item to this line and look no further. We have achieved a correct result. However allocating a manual selection priority to all lines would prove tedious. Consequently MI offers an automatic facility to default the selection priority based upon an analysis of how specific the definition of the selections are. Again in the above example, Current Accounts for Banks is a more specific selection than just Current Accounts and records would be compared to this first. The same result would have been achieved as when the priorities were allocated manually. MIs determines how specific a selection is by analysing the number of different operands and fields used in the selection. This process is described in more detail below. MI holds the sequence in which it will check lines in the field int.sel.priority on mi.line. This field consists of two parts: the selection.priority as input by the user and an analysis of the relative specificity of the selection parameters for the line. MI determines the relative priority of lines by analysing the mi.selection records that the mi.line record refers to. It examines three factors: - The number of different source fields used in the selection records - The number of different operands used on those fields - The relative priority of those operands (see the table below) For example if MI is presented with two lines with the following selections: Line 1 - PRODCAT EQ 1000 Line 2 - PRODCAT EQ 1000 AND CURRENCY EQ USD it will assign data to line 2 if it matches. Only if it does not match here will it check against line 1. MI holds its assessment of the relative priority of each line in the field internal selection priority on the mi.line record. Validation Rules: No input. System generated field.

        value - The value to set
        Returns:
