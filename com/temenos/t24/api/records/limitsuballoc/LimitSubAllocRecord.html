      • addOverride

        public void addOverride(java.lang.CharSequence param)
        Appends the specified Override to the end of the collection.
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Contains all the override messages which the User agreed to during input. During the validation stage of the suballocation, the system may provide the User with a series of screen override messages to indicate an anomaly. In the event of the User confirming the override, the full text of the actual override message will be stored, for reference purposes only, in this multivalue field. For example, if the User tries to suballocate more than what is actually available then, when the transaction is validated, a warning bell will sound and a message will appear at the foot of the screen which reads as follows: 'EXCESS INCREASED TO ...........' This will draw the User's attention to the amount suballocated which may be a mistake and can then be corrected. On the other hand, the input may be correct and so the User confirms that this amount is intentional. If, subsequently, the suballocation record is either fully displayed or printed, then all override messages will be present. Validation Rules: (Internal field) Multivalue field.

        Parameters:
        param - the Override to append
        Returns:

        public TField getAutoRestoreAlloc()
        Returns the AutoRestoreAlloc

        Field description: Defines whether the sub-allocation is returned automatically or dependent on if the allocation is required to cover an excess. Entering or defaulting the value Y indicates to the system that the allocation is automatically returned on expiry. A value of N must be entrered manually to instruct the system to check the LIMIT that is receiving the sub-allocation and to return any or all unused Limit if possible. Validation Rules: 1 character mandatory inout of Y or N Defaults to Y

      • getExpiryDate

        public TField getExpiryDate()
        Returns the ExpiryDate

        Field description: Defines the date on which the Suballocation ceases to be effective. If a Suballocation is to last for 1 day only, the Effective Date and the Maturity Date must be the same date. Note:During end of day batch run, on the expiry date or on the last working day before the expiry date (in case if expiry date falls on a holiday), the sub allocation will be reversed. Any excess caused by this reversal will be reported in the normal way. If field AUTO.RESTORE.ALLOC is set to N then the expiry date may be extended by one day until the suballocation can be returned withoout creating an excess Limit. Validation Rules: 6 to 11 type D (date format in range 1950-2049) characters. (Optional input. Default to Effective Date.) The date input must be greater than or equal to the Effective Date (Field 26). The date input must be greater than or equal to the Run date.

      • getOnlineLimitAmt

        public TField getOnlineLimitAmt()
        Returns the OnlineLimitAmt

        Field description: Defines the ONLINE LIMIT to be Suballocated. Validation Rules: 1 to 14 type AMT (Standard Amount Format) characters. (Mandatory input; No default value). The position of the decimal point must be as given by the Currency table entry for the Currency shown in Field 21. The ONLINE LIMIT Amount on the Limit record(s) must be present. When there is a valid Limit facility defined in SUBALLOCATE FROM (Field 1), the system will check that the amount input in this field is not greater than the total of ONLINE LIMIT + SUBALLOCATION TAKEN - SUBALLOCATION GIVEN - COMMITMENT AMOUNT of this Limit facility. In case a conversion of amount the currency table middle rate will be used for comparison purposes. A tolerance of 10% will be accepted. If this amount is greater then the available amount of the Limit, an override will be required.

      • getSubAllocateFrom

        public TField getSubAllocateFrom()
        Returns the SubAllocateFrom

        Field description: Defines the credit line, or sub limit or branch which is Giving the Suballocation. Information about the origin of a Suballocation must be input to the system. Validation Rules: a 1 - 6 numeric characters Customer Number or 3-10 type MNE character Customer Mnemonic. b) '.' c) 3 - 7 numeric character Limit Reference code. d) '.' (Mandatory if Serial Number entered, otherwise not allowed). e) 1-2 numeric character Serial Number in the range 1 to 99. (Optional input, default is 1.) (Mandatory Input.) If a customer number is input, a valid record must exist on the Customer file with an ID equal to the input. Any Limit Reference number or Serial Number input must also exist for the corresponding Liability number. At least SUB ALLOCATE FROM or SUB ALLOCATE TO must be a credit line or Sub Limit ID. If a valid limit is input in this field, an override will be requested if this Limit has been flagged as 'Not Available'. Sharing group limits can be sub allocated. Sub allocation involving group limits can be any of the below,

        From a group limit to an individual limit.

        From an individual limit to group limit.

        From a group limit to a group limit.

        Parameters:
        Returns:
        TField - The SubAllocateFrom

      • getSubAllocateTo

        public TField getSubAllocateTo()
        Returns the SubAllocateTo

        Field description: Defines the credit line, or Sub Limit or Branch which is Receiving the Suballocation. Information about the destination of a Suballocation must be input to the System. Validation Rules: a) 1-6 numeric character Customer Number or 3-10 type MNE character Customer Mnemonic. b) '.' c) 3 - 7 numeric character Limit Reference code. d) '.' (Mandatory if Serial Number entered, otherwise not allowed.) e) 1-2 numeric character Serial Number in the range 1 to 99. (Optional input, default is 1.) (Mandatory input.) If a customer number is input, a valid record must exist on the Customer file with an ID equal to the input. If a credit line or sub limit ID is input, it must be a valid ID on the Limit file. The Limit reference portion of this will be used to check the Limit Reference file for the field GET SUB ALLOC. If the SUB ALLOCATE FROM is also a credit line or Sub Limit, then the reference portion of this must be recorded in the GET SUB ALLOC field. At least SUB ALLOCATE FROM or SUB ALLOCATE TO must be a credit line or Sub Limit ID. Sharing group limits can be sub allocated. Sub allocation involving group limits can be any of the below,

        From a group limit to an individual limit.

        From an individual limit to group limit.

        From a group limit to a group limit.

        Returns:

        Set the specified Override to a specific position in collection.
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Contains all the override messages which the User agreed to during input. During the validation stage of the suballocation, the system may provide the User with a series of screen override messages to indicate an anomaly. In the event of the User confirming the override, the full text of the actual override message will be stored, for reference purposes only, in this multivalue field. For example, if the User tries to suballocate more than what is actually available then, when the transaction is validated, a warning bell will sound and a message will appear at the foot of the screen which reads as follows: 'EXCESS INCREASED TO ...........' This will draw the User's attention to the amount suballocated which may be a mistake and can then be corrected. On the other hand, the input may be correct and so the User confirms that this amount is intentional.
