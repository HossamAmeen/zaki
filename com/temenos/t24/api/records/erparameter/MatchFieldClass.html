      • getMatchField

        public TField getMatchField()
        Returns the MatchField

        Field description: For matching purposes the fields ACCOUNT.ID, VALUE.DATE and CCY.AMOUNT must be entered here so matching can be done. It is also possible to refine the processing so a match needs to check either or both of the TRANSACTION.REFERENCE or ORD.CUSTOMER. Tolerance checking is allowed on the CCY.AMOUNT and/or VALUE.DATE fields. Associated with the multi-value set from field 5 to 12 and associated with sub-value set from field 13 to 14 If the Exp.Funds.Type is mentioned as ERR, then the fields ACCOUNT.ID and CCY.AMOUNT are mandatory. Validation Rules: ACCOUNT.ID, VALUE.DATE and CCY.AMOUNT must be entered for Exp.Type ER and EP ACCOUNT.ID, and CCY.AMOUNT must be entered for Exp.Type ERR

      • getTolerance

        public TField getTolerance()
        Returns the Tolerance

        Field description: This will be the tolerance for the related multi-value, explained above. If there is no tolerance entered then there must be an exact match on the related field. There will be some fields where having a tolerance is pointless, such as narrative, but this will be up to the bank to decide and no controls will be added for that here. Currently, the system can handle tolerances on CCY.AMOUNT and the VALUE.DATE. When the tolerance is defined, the system will first try to match with the exact keys. If no match is found then the tolerances defined are used for calculating/determining the tolerance keys on the record to be matched. The matching process is repeated to find the tolerance match. It should be noted that the match on the ACCOUNT.ID, VALUE.DATE and CCY.AMOUNT are mandatory for EXP.FUND.TYPE: ER and EP and for EXP.FUND.TYPE : ERR, the match on the ACCOUNT.ID, CCY.AMOUNT are mandatory Validation Rules: CCY.AMOUNT The tolerance on amount is entered as a percentage. This is a plus and minus tolerance with a maximum permitted tolerance of 10. The expected and receipts will be matched when the amount is within the tolerance limits, if specified. The lower and higher tolerance amount is calculated on the expected amount. The amount on received record should be within these limits. In the event of their being more than one record matched within the limits the first record from the selected list is matched, this may not be the closest matched record. VALUE.DATE The tolerance for the VALUE.DATE is in number of days and expected funds with receipt funds will be matched when the value dates are with in the tolerance date. This is plus and minus number of calendar days. A maximum value of plus/minus 9 days is permitted. When there is a match under these conditions then the amount expected and received will be shown on the different value dates on the ACCOUNT file on the ER.VALUE.DATE and ER.BALANCE. The receipts will be in negative where as the expected will be positive. If it is required to correct the ER.BALANCE in the system for a particular account, it will be necessary manually create opposite records; one for the expected funds and another for the receipt funds.

        Returns:
