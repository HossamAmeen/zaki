        public TField getApplication()
        Returns the Application

        Field description: Specifies which Application may be used to access records belonging to the Company specified in the related COMPANY RESTR (Field 16). This field is used, together with the associated Fields 16 and 18-23 to specify precisely which records in which Applications can be accessed by the User using which Versions and Functions. It must be the name of an existing Application, the name of an existing USER.SMS.GROUP preceded by the '@' sign (in which case further input in fields 18.1 to 23.1 will not be allowed), ALL.PG to allow access to all Applications or ALL.PG (program type) to allow access to all Applications of a certain type (see below). The following program types may be used in conjunction with ALL.PG (e.g. ALL.PG.H) to give access to a particular type of program:- Program Type Meaning H Programs for which the standard 3 files exist - Live, Unauthorised and History, e.g. CUSTOMER. U Programs for which Live and Unauthorised files exist, but no History, e.g. VERSION. L Programs for which only a Live file exists (updated automatically by the system) e.g. STMT.ENTRY (only L, S and P functions can be used with this type of program). M Programs which are designed to do something other than processing the standard T24 functions using a specific file e.g. PASSWORD.ENABLE, End Of Day Batch programs. T Programs which handle automatically updated CONCAT files, e.g. CUSTOMER.ACCOUNT (only L, S and P functions can be used with this type of program). Note: The following 3 type M programs are available to all users and cannot be restricted:- SIGN.ON SIGN.OFF PASSWORD Example 1: To give a User access to the records of only one Company (GB-001-0000) but allow access to all records for this Company using all Functions (except auditor review) and all Versions of all Applications, enter the following: Field 4.1 (COMPANY) GB-001-0000 Field 16.1(COMPANY CODE) Field 17.1(APPLICATION) ALL.PG Field 18.1(VERSION) Field 19.1(FUNCTION) ALL Note: Fields 20-23 are used to specify particular records or types of records which may be accessed within an Application. See FIELD NO (Field 20) for examples. Example 2: To give a User full access (except auditor review) to the records of Company GB-001-0000 and to give limited access to the records of Company GB-001-0001, only allowing the List and Display functions for Customer Application. Field 4.1(COMPANY) GB-001-0000 Field 4.2(COMPANY) GB-001-0001 Field 16.1(COMPANY.CODE) GB-001-0000 Field 17.1(APPLICATION) ALL.PG Field 18.1(VERSION) Field 19.1(FUNCTION) ALL Field 16.2(COMPANY.CODE) GB-001-0001 Field 17.2(APPLICATION) CUSTOMER Field 18.2(VERSION) Field 19.2(FUNCTION) L S Example 3: To give a User access to the records of only one Company (GB-001-0000), allowing him to List Display, Input and Copy Customer records and to List and Display records for all other Applications: Field 4.1(COMPANY) GB-001-0000 Field 17.1(APPLICATION) ALL.PG Field 19.1(FUNCTION) L S Field 17.2(APPLICATION) CUSTOMER Field 19.2(FUNCTION) L S I C Example 4: To prevent a USER from using any type M Applications, but allow him to List and Display records in any other Applications: Field 4.1(COMPANY) GB-001-0000 Field 16.1(COMPANY CODE) Field 17.1(APPLICATION) ALL.PG.H Field 19.1(FUNCTION) L S Field 17.2(APPLICATION) ALL.PG.U Field 19.2(FUNCTION) L S Field 17.3(APPLICATION) ALL.PG.L Field 19.3(FUNCTION) L S Field 17.4(APPLICATION) ALL.PG.T Field 19.4(FUNCTION) L S Validation Rules 1. 1-30 type SSS (uppercase alpha) characters, Application name.or 2.         Field description: Specifies the Company to which the User must be Signed On in order to use the Applications, Versions and Functions specified in the related Fields 17 to 23. This field is used, together with the associated Fields 17-23, to specify for each Company to whose records this User has access, precisely which records in which Applications can be accessed using which Versions and Functions. The group of Fields, 16-23 can be repeated up to 999 times. Refer to APPLICATION (Field 17) for examples. Any files for which standard or special file mnemonics have been specified in the Company record can only be accessed by Users who have Signed On to that Company (or have used the PASSWORD Application to access that Company). Company Mnemonic characters.Multi-value field associated with Fields 17-23.Mandatory input. No Default Value.It must be a Company Code from the COMPANY.CODE field of this record.

        public TField getDataFrom()
        Returns the DataFrom

        Field description: Specifies the value, or start of a range of values, with which the contents of the field indicated in FIELD NO (Field 20) are to be compared. 1-35 type A (alphanumeric) characters. OR Up to 55 characters (type ANY) formatted in the following way: (i) Up to 30 characters (type ANY) which must be a valid entry on the FILE.CONTROL file, followed by, (ii) A '>' (greater than) sign, followed by, (iii) Up to 18 characters (type A) which must be a valid entry on the multi-value field SYS.FIELD.NAME on the STANDARD.SELECTION record for the file specified in (i). Multi-value field associated with Fields 16-21 and 23. (Mandatory input if FIELD NO is entered, otherwise input is not allowed.) This field specifies either a single value or the lower limit of a range with which the contents of the field indicated in FIELD NO (Field 20) are compared. Note: The values input in this field and the next one must be in agreement with the Format rules of the element specified in Field 20.1. If, for example, the input in Field 20 corresponds to a Customer Mnemonic, a minimum input of 3 characters will be required in this field. When defining the enquires a user can run, you can also 'turn off' all SMS checks for a particular enquiry. This is useful when the enquiry presents non sensitive information from sensitive files. I.e you may wish to show customer and account static information (address, account titles etc ) but not show the balances. As the enquiry itself controls the presentation there may be no need for any additional security. To suppress SMS you must append *NOSMS to the name of the enquiry. E.G. Field 17.1 (APPLICATION) ENQUIRY.SELECT Field 19.1(FUNCTION) I Field 20.1(FIELD.NO) 0 Field 21.1(DATA.COMPARISON) EQ Field 22.1(DATA.FROM) CUSTOMER.STATIC*NOSMS This mechanism works only with ENQUIRY.SELECT and must be entered in the format detailed above. Validation Rules See Details for format.Input must begin and/or end with 3 dots when DATA.COMPARISON = 'LK' or 'UL'.

      • getFieldNo

        public TField getFieldNo()
        Returns the FieldNo

        Field description: Specifies, when used together with Fields 21-23, precisely which records may be accessed in the Application specified in the corresponding APPLICATION (Field 17.) A record may only be accessed if the field indicated here has a value allowed in the corresponding Fields 21-23. 0-3 numeric characters. OR Up to 55 characters (type ANY) formatted in the following way: (i) Up to 30 characters (type ANY) which must be a valid entry on the FILE.CONTROL file, followed by, (ii) A '>' (greater than) sign, followed by, (iii) Up to 18 characters (type A) which must be a valid entry on the multi-value field SYS.FIELD.NAME or USR.FIELD.NAME on the STANDARD.SELECTION record for the file specified in (i). Eg: FOREX>COUNTERPARTY It is recommended that field names and not field numbers are specified. If field numbers are specified then when a file layout changes you will be forced to manually change all user profiles affected. Multi-value field associated with Fields 16-19 and 21-23. (Optional input. No input means all records may be accessed, regardless of contents.) For multi-value fields, only the field number may be specified, not the entry number, e.g. 20, not 20.1, all entries in the multi-value field will be checked. The System displays the name of the field indicated. Example 1: To allow a User to See and List all Accounts records, but only Input (or change), Copy or Delete Accounts in pounds sterling. Field 17.1(APPLICATION) ACCOUNT 19.1(FUNCTION) L S 17.2(APPLICATION) ACCOUNT 19.2(FUNCTION) I C D 20.2(FIELD NO.) ACCOUNT>CURRENCY (currency) 21.2(DATA COMPARISON) EQ (equal) 22.2(DATA FROM) GBP Example 2: To allow a User access to Account records only if they have Currency equal GBP and Category not in the range 01-001 to 01-099. Field 17.1(APPLICATION) ACCOUNT19.1(FUNCTION) ALL 20.1(FIELD.NO) ACCOUNT>CURRENCY (Currency) 21.1(DATA.COMPARISON) EQ (equal) 22.1(DATA.FROM) GBP 17.2(APPLICATION) ACCOUNT 19.2(FUNCTION) ALL 20.2(FIELD.NO) ACCOUNT>CATEGORY (Category) 21.2(DATA.COMPARISON) NE (not equal) 22.2(DATA.FROM) 01-001 23.2(DATA.TO) 01-099 Example 3: To allow a User to Authorise Data Capture records, but only if they do not contain an override. Field 17.1(APPLICATION) DATA.CAPTURE 19.1(FUNCTION) A 20.1(FIELD.NO) DATA.CAPTURE>OVERRIDE (Override) 21.1(DATA COMPARISON) LT (Less Than) 22.1(DATA.FROM) "" (lowest possible value) Note: It is not possible to prevent a User from entering on override during input, but Authorization can be restricted, as above, to ensure that any overrides are checked by a more senior person. "" is the lowest value a field can have, so specifying LT (Less Than) "" ensures that no override may be present. Example 4: Restricting FOREX transactions to a counterparties that belong to a specific account officer. Field 17.1(APPLICATION) FOREX 19.1(FUNCTION) I20.1(FIELD.NO) CUSTOMER>ACCOUNT.OFFICER 21.1(DATA COMPARISON) EQ22.1(DATA.FROM) 100 Note: As you can see the field specified does not have to belong to the application. If the system can link the APPLICATION to the field then the SMS restriction will be applied. In this example the COUNTERPARTY field on the FOREX file provides the link to the CUSTOMER file. Hence when the transaction is input T24 reads the CUSTOMER file using the COUNTERPARTY as the key and checks that the ACCOUNT.OFFICER is 100. You do not have to define that the COUNTERPARTY as the key to CUSTOMER file this is done automatically based on the RELATED.FILE stored in the STANDARD.SELECTION record for FOREX. Note: If you wish to test a value in a related file a consderable process load can be incurred when the user lists the file. This can be relieved by defining the related field as a J-descriptor user field on the STANDARD.SELECTION record and using that. Thus in the example above define CUST.OFFICER as type J, CUSTOMER>ACCOUNT.OFFICER and then specify the FIELD.NO as FOREX>CUST.OFFICER. Validation Rules See Details for format.It must not be higher than the number of fields in records of the Application specified in APPLICATION (Field 17.)Input in this field will not be allowed when 'N' has been input in 'FUNCTION' (Field 19.1).

        public TField getFunction()

        Field description: Specifies the Functions which may be used to access records of the Application entered in the corresponding Field 17. This field is used, together with the associated Fields 16-18 and 20-23 to specify precisely which Functions the User may be used to access which records in which Applications. Valid Function codes are: A 2 C D E H I L P Q R S V If NOT is entered, plus one or more Function codes, the System converts it to a list of all the Functions which are allowed. (See Fields 17, 18 and 20 for examples.) Note: Even if the function 'C' (copy) is not allowed, the user will still be able to transfer the ID (together with its enrichment) of a selected record when using the '!' facility during an input function. Validation Rules 0-29 type S (uppercase alpha or numeric) characters: 1. Function codes separated by spaces or 2. NOT + one or more Function codes or 3. N (No Functions allowed)or 4. ALL (All Functions allowed). Multi-value field associated with Fields 16-18, 20-23. Mandatory input except if a USER.SMS.GROUP was defined in field 18.1 in which case input is not allowed.No Default Value. If Function 2 (2nd Authorization) is allowed, but not A (1st Authorization) the User is allowed to enter Function Code A, a Security Violation is generated if the record requested is not ready for 2nd Authorization.

      • getVersion

        public TField getVersion()
        Returns the Version

        Field description: Specifies a particular Version which may be used to access records for the Application entered in the corresponding APPLICATION (Field 17.) This field is used, together with the associated Fields, 16, 17 and 19-23, to specify precisely which records in which Applications may be accessed by the User, using which Versions and Functions. * and ? can be used in this field , to specify more than one Version. They act as wild card characters ,where * replaces any number of characters including null, numbers, alphabets etc. and ? replaces one character including null, numbers, alphabets etc. The version field in can have the value for eg : ,SAVE* , which allows the user to access Versions of ACCOUNT applications for eg : ACCOUNT,SAVESTUDENT , ACCOUNT,SAVERESIDENT , ACCOUNT,SAVESALARIED, etc The version field can have the value for eg : ,SAVE1000?00,which allows the user to access versions of ACCOUNT application for eg: SAVE1000100, SAVE1000200, SAVE1000300 , etc Note: Versions of Applications may be used to change the screen layout or the rules for Input, Validation and Authorization. Example 1: To allow a User to List and See all Accounts except for those with Account Officer = 1234 using the standard version of Account, and to List and See all Accounts for Account Officer = 1234 only, using a special Version Account,99: Field 17.1(APPLICATION) ACCOUNT 18.1(VERSION) ,0 19.1(FUNCTION) L S 20.1(FIELD NO) 9 21.1(DATA COMPARISON) NE 22.1(DATA FROM) 1234 17.2(APPLICATION) ACCOUNT 18.2(VERSION) ,99 19.2(FUNCTION) L S 20.2(FIELD NO) 9 21.2(DATA COMPARISON) EQ 22.2(DATA FROM) 1234 Example 2 : To allow a User to Input and See all Account version with name starting with SAVINGS followed by any character: Field 17.1(APPLICATION) ACCOUNT 18.1(VERSION) ,SAVINGS* 19.1(FUNCTION) I S For more examples of Fields 16-23 see Fields 17 and 20. Validation Rules 1. 0-50 type A (alphanumeric) characters beginning with ','. or 2. ',0' to allow only the standard version. or 3. No input to allow all Versions. Multi-value field associated with Fields 16, 17 and 19-23. (Optional input except if a USER.SMS.GROUP was defined in field 17.1 in which case input is not allowed. If not entered, access to all Versions is allowed.Special character * will replace any number of characters and ? replaces one character. Also ? after * and two occurances of * are not allowed.
