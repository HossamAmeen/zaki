      • getValueDatePrd

        public TField getValueDatePrd()
        Returns the ValueDatePrd

        Field description: Identifies for the TELLER, FUNDS TRANSFER and DATA.CAPTURE Application, the Default Value Date which will be assigned to all transactions where this Transaction code has been defined. This Default Value Date can always be overridden by entering the Value Date directly at transaction level. If Y, or N(O) are entered the inputs (a) to (f) are not allowed. (a), (bb), (c) specify the number of calendar or working days to be added/subtracted to/from the Run Date in order to obtain the Default Value Date as follows: (a) + If days are to be added to the Run Date. The '+' sign should normally be used for credit Entries. - If days are to be subtracted from the Run Date. The '-' sign should normally be used for Debit entries. (bb) 01-99Indicates the number of days to be added or subtracted. (c) C If (bb) are calendar days, W If (bb) are working days. If it is necessary to add/subtract any other days to those specified in (bb) before obtaining the Default Value Date then (d), (ee), (f) are used as follows: (d) As for (a) previously. (ee) As for (bb) previously. (f) C If ee are calendar days. W If ee are working days. It must be noted that the day type ('C' or 'W') can only be used once in an equation. Note: It is important to realise that the conditions defined in this field are only Applicable for the applications named above. All other Applications have their Value Date conditions defined by their own tables. This field allows a Default Value Date to be specified for a TELLER , FUNDS.TRANSFER and DATA.CAPTURE transaction either, equal to the Run Date, or the Run Date plus or minus a number of working or calendar days. This prevents the operator from having to key the Value Date on transactions repeatedly, where a default can be applied. It can be overridden at Transaction level when required. For the DATA CAPTURE application field only this default is also used for the exposure date as well as the value date. Example: +03W will mean that the default value date is 3 working days forward from the Run Date. +01C + 02W will mean that the default value date is 1 calendar day + 2 working days forward from the Run date. Validation Rules: Y If the Default Value Date is to be the Run Date. or N(O) If a Default Value Date is not to be generated automatically. Value Date must then be input manually when Transaction Code is used. or (a) (bb) (c) (d) (ee) (f) As defined in the example below Run Date is the date of the data being processed which may not always be today's date. If different, the Run Date is displayed next to the User ID on the screen. The same day type ('C' or 'W') can only be used once in an equation ie. +01C + 02C will not be accepted. When both working and calendar days are entered, the sign must be the same i.e. +01C - 02W will not be accepted. It is not necessary to enter the '+-' signs. If omitted it will always default to '+' This default overtakes any previous default set up i.e. takes presidence over the transaction code default.

        Returns:
        TField - The ValueDatePrd

      • setValueDatePrd

        public void setValueDatePrd(java.lang.CharSequence value)
        Set the specified value for ValueDatePrd
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Identifies for the TELLER, FUNDS TRANSFER and DATA.CAPTURE Application, the Default Value Date which will be assigned to all transactions where this Transaction code has been defined. This Default Value Date can always be overridden by entering the Value Date directly at transaction level. If Y, or N(O) are entered the inputs (a) to (f) are not allowed. (a), (bb), (c) specify the number of calendar or working days to be added/subtracted to/from the Run Date in order to obtain the Default Value Date as follows: (a) + If days are to be added to the Run Date. The '+' sign should normally be used for credit Entries. - If days are to be subtracted from the Run Date. The '-' sign should normally be used for Debit entries. (bb) 01-99Indicates the number of days to be added or subtracted. (c) C If (bb) are calendar days, W If (bb) are working days. If it is necessary to add/subtract any other days to those specified in (bb) before obtaining the Default Value Date then (d), (ee), (f) are used as follows: (d) As for (a) previously. (ee) As for (bb) previously. (f) C If ee are calendar days. W If ee are working days. It must be noted that the day type ('C' or 'W') can only be used once in an equation. Note: It is important to realise that the conditions defined in this field are only Applicable for the applications named above. All other Applications have their Value Date conditions defined by their own tables. This field allows a Default Value Date to be specified for a TELLER , FUNDS.TRANSFER and DATA.CAPTURE transaction either, equal to the Run Date, or the Run Date plus or minus a number of working or calendar days. This prevents the operator from having to key the Value Date on transactions repeatedly, where a default can be applied. It can be overridden at Transaction level when required. For the DATA CAPTURE application field only this default is also used for the exposure date as well as the value date. Example: +03W will mean that the default value date is 3 working days forward from the Run Date. +01C + 02W will mean that the default value date is 1 calendar day + 2 working days forward from the Run date. Validation Rules: Y If the Default Value Date is to be the Run Date. or N(O) If a Default Value Date is not to be generated automatically. Value Date must then be input manually when Transaction Code is used. or (a) (bb) (c) (d) (ee) (f) As defined in the example below Run Date is the date of the data being processed which may not always be today's date. If different, the Run Date is displayed next to the User ID on the screen. The same day type ('C' or 'W') can only be used once in an equation ie. +01C + 02C will not be accepted. When both working and calendar days are entered, the sign must be the same i.e. +01C - 02W will not be accepted. It is not necessary to enter the '+-' signs. If omitted it will always default to '+' This default overtakes any previous default set up i.e. takes presidence over the transaction code default.

        Parameters:
        value - The value to set
        Returns:
        nothing
