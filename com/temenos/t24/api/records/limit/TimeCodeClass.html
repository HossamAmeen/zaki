      • getLimitBand

        public TField getLimitBand()
        Returns the LimitBand

        Field description: Specifies the split of the ONLINE LIMIT amount over the time bands in field 25, TIME CODE. If the limit amount is to be made available over specified maturity periods (field 25, TIME CODE), this field is used to apportion the overall amount across those time bands. It is the amount in field 21, MAXIMUM TOTAL, which should be apportioned. The first multivalue in the set of associated fields (25 to 45) is used to record details of the entire limit amount and its utilisation. The amount in field 26, value 1 should therefore always be the same as the full MAXIMUM TOTAL in field 21. The amounts in field 26, values 2 onwards should add up to this same overall total but indicate the individual amounts available within each time period. In the case of a variable secured limit, the MAXIMUM TOTAL in field 21 will not necessarily be the same as the allowed ONLINE LIMIT amount in field 27. The limit bands are, however, still specified in terms of the MAXIMUM TOTAL. The system automatically apportions the ONLINE LIMIT amount pro-rata. The default value in 26.1 is the MAXIMUM TOTAL defined in field 21. The default value in 26.2 onwards is the difference between this total and the sum of the values occuring in the earlier multivalues. Validation Rules: 1 to 19 type AMT (standard amount format). (Mandatory input when time bands specified, otherwise not allowed; for default value see details) Multivalue field associated with fields 25 to 45. No input is allowed unless field 75, FX.OR.TIME.BAND, is 'FX' or 'TB'. No input is allowed unless time bands are specified (field 25, TIME CODE). The first value must equal the MAXIMUM TOTAL in field 21. Subsequent values may not exceed the MAXIMUM TOTAL in field 21, nor may their sum.

        Returns:
        TField - The LimitBand

      • getOnlineLimit

        public TField getOnlineLimit()
        Returns the OnlineLimit

        Field description: The limit amount which is actually available for use. The amount available under the limit, against which all system limit checks are performed. The value of the ONLINE LIMIT is calculated by the system as described below. If the limit is an unsecured limit, a sub-limit or a secured but fixed limit, the value in this field is always equal to the value in field 21, MAXIMUM TOTAL. If the limit is a variable secured limit, the value in this field is equal to the lower of (a) the MAXIMUM TOTAL or (b) the sum of the SECURED AMT's in field 24 plus the MAXIMUM UNSECURED in field 20. Note: If time bands are in effect (field 25, TIME CODE), the ONLINE LIMIT amount will reflect the split of the available limit across these periods. In the case of an unsecured limit, a sub-limit or a fixed secured limit, this field will contain the same values as field 26, LIMIT BAND. In the case of a variable secured limit, this field will contain either the same values as field 26, LIMIT BAND or, if the total ONLINE LIMIT amount is lower than the MAXIMUM TOTAL in field 21, the ONLINE LIMIT amount apportioned pro-rata across the time bands in line with the specified LIMIT BAND split. Validation Rules: 1 to 19 type AMT (standard amount format). (No input, system-generated field) Multivalue field associated with fields 25 to 45.

      • getTimeCode

        public TField getTimeCode()
        Returns the TimeCode

        Field description: Specifies time sub-divisions of a Limit, each of which is available for a different maturity period. The theory behind the use of this field is that the further forward the maturity date of a transaction, the greater the associated risk. The Limit is therefore approved on the basis that the whole Limit is available for the most immediate time slice. The amount available then decreases as the maturity time slice increases. Example: A Money Market line of $100 million may be sub-divided into $60 million for maturity within 30 days, $30 million within 90 days, and the remaining $10 million within 180 days. To enter the above limit, the overall $100 million should be entered with a 'blank' time code. If a transaction exceeds the specified period (in this case 180 days) and the referral message is overriden, the system will reduce the amount available in the latest band specified (180 days). It should be noted that when a band is not fully utilised it becomes available for use at lower levels. In the example, if only $8 million were used in the 180 day period, $32 million would be available for 90 day contracts. Validation Rules: Up to 4 numeric characters in the range 1-9999.       • setTimeCode

        public void setTimeCode(TField value)
        Set the specified value for TimeCode
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Specifies time sub-divisions of a Limit, each of which is available for a different maturity period. The theory behind the use of this field is that the further forward the maturity date of a transaction, the greater the associated risk. The Limit is therefore approved on the basis that the whole Limit is available for the most immediate time slice. The amount available then decreases as the maturity time slice increases. Example: A Money Market line of $100 million may be sub-divided into $60 million for maturity within 30 days, $30 million within 90 days, and the remaining $10 million within 180 days. To enter the above limit, the overall $100 million should be entered with a 'blank' time code. The associated multi-valued fields should be expanded to enter the sub-division as follows: TIME.CODE COMMITTED.AMOUNT 30 $60 million 90 $30 million 180 $10 million The example above shows that the bands must be entered in ascending order. If a transaction exceeds the specified period (in this case 180 days) and the referral message is overriden, the system will reduce the amount available in the latest band specified (180 days). It should be noted that when a band is not fully utilised it becomes available for use at lower levels. In the example, if only $8 million were used in the 180 day period, $32 million would be available for 90 day contracts. Validation Rules: Up to 4 numeric characters in the range 1-9999. Multivalue field associated with fields 26.1 to 45.1. (Optional input. No default value.) Input can only be made where the FX.OR.TIME.BAND field on the associated Limit Reference record has a value of 'TB', 'FX' or 'BD'. In other words the definition of time bands will be allowed for all types of Limit reference except the ones which are related to the application 'ACCOUNT'. The total input to occurrences with an associated Time Code must equal the amount input to the occurrence without a Time Code. The time codes, when defined, must be entered in ascending order. Only Single time band allowed for group limits and Multiple time bands are restricted.

        value - The value to set
