        public void addFileSorting(TField param)
        Appends the specified FileSorting to the end of the collection.
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Specifies the order, in which the OFS messages are sorted, before pumping it into T24 system. Before creating the main file record, the dependent file records are created. In case of VERSION deployment from design studio, the received xml can have dependent file records. The EB.API, PGM.FILE and EB.DICTIONARY are dependent files for VERSION. Hence the dependent file records are created first and then VERSION record is created. For this purpose, the sorting order of OFS messages must be specified in this field. Validation Rules:

        Must be a valid application name.

        The dependent files should be specified before the mainfile.

        Parameters:
        param - the FileSorting to append
        Returns:

      • getVersionName

        public TField getVersionName()
        Returns the VersionName

        Field description: Specifies the application and version, by which the records are created via OFS. If DS.PARAM.ID = VERSION, then VERSION.NAME can have any existing version id of the application called 'VERSION'. If DS.PARAM.ID = ENQUIRY, then VERSION.NAME can have any existing enquiry id of the application called 'ENQUIRY'. For e.g. To create a record in VERSION application, the value is specified as VERSION,DS which will be used in OFS message. The 'DS' is the pgm version for other files, which are specified in FILE.SORTING field. If the FILE.SORTING field is specified with the files EB.API, PGM.FILE and EB.DICTIONARY then the version installer will use the versions such as 'EB.API,DS' , 'PGM.FILE,DS' and 'EB.DICTIONARY,DS' respectively to form OFS messages. Validation Rules:

        If DS.PARAM.ID = VERSION (installer type), then the application name in VERSION.NAME field should be VERSION.

        If DS.PARAM.ID = ENQUIRY (installer type), then the application name in VERSION.NAME field should be ENQUIRY. This field cannot be blank if valid application name given as @ID

        Must be a valid entry in VERSION table.

        Allows maximum of 35 alphanumerics.

        Parameters:
        Returns:
        TField - The VersionName

        Set the specified value for VersionName
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Specifies the application and version, by which the records are created via OFS. If DS.PARAM.ID = VERSION, then VERSION.NAME can have any existing version id of the application called 'VERSION'. If DS.PARAM.ID = ENQUIRY, then VERSION.NAME can have any existing enquiry id of the application called 'ENQUIRY'. For e.g. To create a record in VERSION application, the value is specified as VERSION,DS which will be used in OFS message. The 'DS' is the pgm version for other files, which are specified in FILE.SORTING field. If the FILE.SORTING field is specified with the files EB.API, PGM.FILE and EB.DICTIONARY then the version installer will use the versions such as 'EB.API,DS' , 'PGM.FILE,DS' and 'EB.DICTIONARY,DS' respectively to form OFS messages.
