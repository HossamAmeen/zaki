      • getRerunCancel

        public TField getRerunCancel()
        Returns the RerunCancel

        Field description: If on checking the customer records after a redemption detail record has been authorised the user discovers an incorrect record he has the facility to 'rerun' the transaction process for that redemption transaction. But this will only take effect at E.O.D. Likewise a cancel feature is provided for all authorised records. On checking customer records a mistake is discovered the user can do two things: 1) Correct the clients position and enter 'R' for rerun in the redemption detail record; which will recalculate, using the corrected balances - but only at End of Day Routine. 2) He may decide to cancel the transaction altogether in which case he will enter 'C' for cancel at the redemption detail record - which will cancel the customer records - but again only at End of Day routine. If the user wishes to pass the redemption transaction again - he must use a different 'redemption type' in order to create a unique transaction key. It is suggested therefore that the user set up a redemption type say 'CORR' - for correction entry which would be identical for purposes of creating the same customer records, yet different for purposes of creating the Redemption Detail record. Validation Rules: Enter R or C only allowed.

        Returns:
        TField - The RerunCancel

        Field description: If on checking the customer records after a redemption detail record has been authorised the user discovers an incorrect record he has the facility to 'rerun' the transaction process for that redemption transaction. On checking customer records a mistake is discovered the user can do two things: 1) Correct the clients position and enter 'R' for rerun in the redemption detail record; which will recalculate, using the corrected balances - but only at End of Day Routine. 2) He may decide to cancel the transaction altogether in which case he will enter 'C' for cancel at the redemption detail record - which will cancel the customer records - but again only at End of Day routine. If the user wishes to pass the redemption transaction again - he must use a different 'redemption type' in order to create a unique transaction key. It is suggested therefore that the user set up a redemption type say 'CORR' - for correction entry which would be identical for purposes of creating the same customer records, yet different for purposes of creating the Redemption Detail record.
