      • getApplication

        public TField getApplication()
        Returns the Application

        Field description: Specifies which Applications are to be included in the group. This field is used, together with the associated Fields 3.1 to 8.1 to specify precisely which records in which Applications can be accessed by the User using which Versions and Functions. It must be the name of an existing Application, ALL.PG to allow access to all Applications or ALL.PG (program type) to allow access to all Applications of a certain type (see below). The following program types may be used in conjunction with ALL.PG (e.g. ALL.PG.H) to give access to a particular type of program:- Program Type Meaning H Programs for which the standard 3 files exist - Live, Unauthorised and History, e.g. CUSTOMER. U Programs for which Live and Unauthorised files exist, but no History, e.g. VERSION. L Programs for which only a Live file exists (updated automatically by the system) e.g. STMT.ENTRY (only L, S and P functions can be used with this type of program). M Programs which are designed to do something other than processing the standard T24 functions using a specific file e.g. PASSWORD.ENABLE, End Of Day Batch programs. T Programs which handle automatically updated CONCAT files, e.g. CUSTOMER.ACCOUNT (only L, S and P functions can be used with this type of program). Note: The following 3 type M programs are available to all users and cannot be restricted:- SIGN.ON SIGN.OFF PASSWORD Example 1: To create a group which will give the User access to all Applications using all Functions (except auditor review) and all Versions of all Applications, enter the following:- Note: Fields 5.1-8.1 are used to specify particular records or types of records which may be accessed within an Application. See FIELD NO (Field 5.1) for examples. Example 2: To create a group which will allow the User to List Display, Input and Copy Customer records and to List and Display records for all other Applications: Example 3: To create a group which will prevent a USER from using any type M Applications, but allow him to List and Display records in any other Applications: Validation Rules: 1. 1-25 type SSS (uppercase alpha) characters, Application name. or 2. ALL.PG to allow all Applications. or 3. ALL.PG (program type) see details. Multivalue field associated with Fields 3.1 to 8.1. (Mandatory input in each set of Fields 2.1 - 8.1).

        public TField getDataFrom()
        Returns the DataFrom

        Field description: Specifies the value, or start of a range of values, with which the contents of the field indicated in FIELD NO (Field 5.1) are to be compared. 1-35 type A (alphanumeric) characters. OR Up to 43 characters (type ANY) formatted in the following way: (i) Up to 24 characters (type ANY) which must be a valid entry on the FILE.CONTROL file, followed by, (ii) A '>' (greater than) sign, followed by, (iii) Up to 18 characters (type A) which must be a valid entry on the multivalue field SYS.FIELD.NAME on the STANDARD.SELECTION record for the file specified in (i). Multivalue field associated with Fields 2.1 to 6.1 and 8.1. (Mandatory input if FIELD NO is entered, otherwise input is not allowed.) This field specifies either a single value or the lower limit of a range with which the contents of the field indicated in FIELD NO (Field 5.1) are compared. Note: The values input in this field and the next one must be in agreement with the Format rules of the element specified in Field 5.1. If, for example, the input in Field 5.1 corresponds to a Customer Mnemonic, a minimum input of 3 characters will be required in this field. When defining the enquires a user can run, you can also 'turn off' all SMS checks for a particular enquiry. This is useful when the enquiry presents non sensitive information from sensitive files. I.e you may wish to show customer and account static information (address, account titles etc ) but not show the balances. As the enquiry itself controls the presentation there may be no need for any additional security. To suppress SMS you must append *NOSMS to the name of the enquiry. E.G. Field 2.1 (APPLICATION) ENQUIRY.SELECT Field 4.1 (FUNCTION) I Field 5.1 (FIELD.NO) 0 Field 6.1 (DATA.COMPARISON) EQ Field 7.1 (DATA.FROM) CUSTOMER.STATIC*NOSMS This mechanism works only with ENQUIRY.SELECT and must be entered in the format detailed above. Validation Rules: See details for format. Input must begin and/or end with 3 dots when DATA.COMPARISON = 'LK' or 'UL'.

        public TField getEndDate()
        Returns the EndDate

        Field description: The date till which the user may have access to the additional functions in TEMP.FUNCTION is specified here. The Input may be all numeric DDMMYY or MMDDYY depending on the INPUT DAY MONTH format in Field 29 of the User record, or alphanumeric, DDMMMYYYY where the day and year are numeric and the month is the first 3 letters of the month, in capitals. If you omit the year and/or the month, the System assumes they are the same as today. Validation Rules: 1-9 type D (date format in range 1950-2049) characters with optional spaces.       • getFieldNo

        public TField getFieldNo()
        Returns the FieldNo

        Field description: Specifies, when used together with Fields 6.1 - 8.1, precisely which records may be accessed in the Application specified in the corresponding APPLICATION (Field 2.) A record may only be accessed if the field indicated here has a value allowed in the corresponding Fields 6.1 - 8.1. 0-3 numeric characters. OR Up to 43 characters (type ANY) formatted in the following way: (i) Up to 24 characters (type ANY) which must be a valid entry on the FILE.CONTROL file, followed by, (ii) A '>' (greater than) sign, followed by, (iii) Up to 18 characters (type A) which must be a valid entry on the multivalue field SYS.FIELD.NAME on the STANDARD.SELECTION record for the file specified in (i). Multivalue field associated with Fields 2.1 to 4.1 and 6.1 to 8.1. (Optional input. No input means all records may be accessed, regardless of contents.)For multivalue fields, only the field number may be specified, not the entry number, e.g. 21, not 21.1, all entries in the multivalue field will be checked. The System displays the name of the field indicated. Example 1: To create a group which allows a User to See and List all Accounts records, but only Input (or change), Copy or Delete Accounts in pounds sterling. Field 2.1 (APPLICATION) ACCOUNT 4.1 (FUNCTION) L S 2.2 (APPLICATION) ACCOUNT 4.2 (FUNCTION) I C D 5.2 (FIELD NO.) 8 (currency) 6.2 (DATA COMPARISON) EQ (equal) 7.2 (DATA FROM) GBP Example 2: To create a group which allows a User access to Account records only if they have Currency equal GBP and Category not in the range 01-001 to 01-099. Field 2.1 (APPLICATION) ACCOUNT 4.1 (FUNCTION) ALL 5.1 (FIELD.NO) 8 (Currency) 6.1 (DATA.COMPARISON) EQ (equal) 7.1 (DATA.FROM) GBP 2.2 (APPLICATION) ACCOUNT 4.2 (FUNCTION) ALL 5.2 (FIELD.NO) 2 (Category) 6.2 (DATA.COMPARISON) NE (not equal) 7.2 (DATA.FROM) 01-001 8.2 (DATA.TO) 01-099 Example 3: To allow a User to Authorise Data Capture records, but only if they do not contain an override. Field 2.1 (APPLICATION) DATA.CAPTURE 4.1 (FUNCTION) A 5.1 (FIELD.NO) 24 (Override) 6.1 (DATA COMPARISON) LT (Less Than) 7.1 (DATA.FROM) " (lowest possible value) Note: It is not possible to prevent a User from entering on override during input, but Authorisation can be restricted, as above, to ensure that any overrides are checked by a more senior person. " is the lowest value a field can have, so specifying LT (Less Than) " ensures that no override may be present. Validation Rules: See details for format. It must not be higher than the number of fields in records of the Application specified in APPLICATION (Field 2.) Input in this field will not be allowed when 'N' has been input in 'FUNCTION' (Field 4.1).

      • getFunction

        public TField getFunction()
        Returns the Function

        Field description: Specifies the Functions which may be used to access records of the Application entered in the corresponding Field 2. This field is used, together with the associated Fields 2.1 - 3.1 and 5.1 - 8.1 to specify precisely which Functions the User may be used to access which records in which Applications. Valid Function codes are: A 2 C D E H I L P Q R S V (A=Authorise, 2=2nd Authoriser, C=copy, D=Delete, E=Exception, H=History restore, I=Input, L=List, P=Print, Q=Auditor review, R=Reverse, S=See, V=Verify) If NOT is entered, plus one or more Function codes, the System converts it to a list of all the Functions which are allowed. Note: Even if the function 'C' (copy) is not allowed, the user will still be able to transfer the ID (together with its enrichment) of a selected record when using the '!' facility during an input function. Validation Rules: 0-29 type S (uppercase alpha or numeric) characters: 1. Function codes separated by spaces or 2. NOT + one or more Function codes or 3. N (No Functions allowed) or 4. ALL (All Functions allowed). Multivalue field associated with Fields 2.1 - 3.1 and 5.1 - 8.1. Mandatory input. No Default Value. If Function 2 (2nd Authorisation) is allowed, but not A (1st Authorisation) the User is allowed to enter Function Code A, a Security Violation is generated if the record requested is not ready for 2nd Authorisation.

        public TField getVersion()
        Returns the Version

        Field description: Specifies a particular Version which may be used to access records for the Application entered in the corresponding APPLICATION (Field 2.1). This field is used, together with the associated Fields, 2.1 and 4.1 - 8.1, to specify precisely which records in which Applications may be accessed by the User, using which Versions and Functions. Note: Versions of Applications may be used to change the screen layout or the rules for Input, Validation and Authorization. Example: 1 To create a group which allows a User to List and See all Accounts except for those with Account Officer = 1234 using the standard version of Account, and to List and See all Accounts for Account Officer = 1234 only, using a special Version Account,99: Field 2.1 (APPLICATION) ACCOUNT 3.1 (VERSION) ,0 4.1 (FUNCTION) L S 5.1 (FIELD NO) 9 6.1 (DATA COMPARISON) NE 7.1 (DATA FROM) 1234 2.2 (APPLICATION) ACCOUNT 3.2 (VERSION) ,99 4.2 (FUNCTION) L S 5.2 (FIELD NO) 9 6.2 (DATA COMPARISON) EQ 7.2 (DATA FROM) 1234 Validation Rules: 1. 0-23 type A (alphanumeric) characters beginning with ','. or 2. ',0' to allow only the standard version. or 3. No input to allow all Versions. Multivalue field associated with Fields 2.1 and 4.1 to 8.1. (Optional input. If not entered, access to all Versions is allowed.)

As the enquiry itself controls the presentation there may be no need for any additional security. To suppress SMS you must append *NOSMS to the name of the enquiry. E.G. Field 2.1 (APPLICATION) ENQUIRY.SELECT Field 4.1 (FUNCTION) I Field 5.1 (FIELD.NO) 0 Field 6.1 (DATA.COMPARISON) EQ Field 7.1 (DATA.FROM) CUSTOMER.STATIC*NOSMS This mechanism works only with ENQUIRY.SELECT and must be entered in the format detailed above. Validation Rules: See details for format. Input must begin and/or end with 3 dots when DATA.COMPARISON = 'LK' or 'UL'.
