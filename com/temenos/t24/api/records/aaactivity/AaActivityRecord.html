      • addActivityType

        public void addActivityType(java.lang.CharSequence param)
        Appends the specified ActivityType to the end of the collection.
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Can accept 'NOLOG' Or Null. This is a No change field. Option NOLOG will ensure that the additional I/O to AA.ACTIVITY.HISTORY is avoided by not storing the activity related information there. If set as NOLOG, the following validations will come into picture.

        Validation Rules: 1. Allowed only for NORR or NOREPLAY activity type defined activities in AA.ACTIVITY.CLASS. 2. If this is for a TRANSACTION activities, it should support DIRECT.ACCOUNTING option which means accounting is handled by the external application and AA just stores the activity reference details. 3. 'NOLOG' activity was restrict to define rule based evaluation in activity restriction property. 4. 'NOLOG' activity was restrict to define activity charge process in activity charge property. 5. 'NOLOG' activity was restrict to define period charge process in periodic charge property.

        Parameters:
        param - the ActivityType to append
        Returns:
        nothing

        public TField getRelatedActivity()
        Returns the RelatedActivity

        Field description: This field is used ONLY whilst simulating an Arrangement scenario and is used only for LENDING, DEPOSIT product lines at the moment.

        For example, when a NEW arrangement is to be simulated for 2 schedules, the schedules could be run only after the arrangement could be disbursed(either fully or partially). But, disbursement is always transaction driven and may not be captured using a AA.SIMULATION.CAPTURE application. To automatically trigger an activity after the current activity(stated in this ID), this field may be utilised. So, a LENDING-NEW-ARRANGEMENT would have LENDING-DISBURSE-COMMITMENT as its RELATED.ACTIVITY. Similarly, automatic repayments on DUEs may be triggered by using this field. The disbursement and repayments generally assume FULL disbursement and payment which may be altered at the AA.SIMULATION.RUNNER to simulate partial disbursements and payments.

        Only an activity belonging to the Activity class stated in its AA.ACTIVITY.CLASS may be stated (RELATED.ACT.CLASS field). User Activities may also be stated here as long as they belong to the core Activity class.

        This field has NO relevance on activities triggered on Live arrangement.

        Validations: Should be a valid entry in AA.ACTIVITY file. Should belong to the Activity class stated in RELATED.ACT.CLASS field of the AA.ACTIVITY.CLASS.
