      • getMsgDisposition

        public TField getMsgDisposition()
        Returns the MsgDisposition

        Field description: Describes the type of processing to be done on this copy of the message. Msg Disposition is not present until after formatting when the Header record is expanded to include a multivalue set of data for each copy of the message. If a problem prevents the formatting of a copy of a message then Msg Disposition in the appropriate multivalue set is set to 'Repair', otherwise it is set to 'Formatted'. Once a Message has been Formatted each copy of the Message is put on the Formatted Queue appropriate to the Carrier. However, if the carrier is 'TELEX', 'TELEXP' or 'TELEXF' and the message is of normal priority, the message will be placed in the batching queue if batching is required (Batching Req on DE.PARM is 'Y') and the message type is to be batched (exists on DE.BATCH.GROUP). Messages to be batched will be placed on the batching file with a Msg Disposition of 'BATCHING' until the Batch is complete. Batches will be automatically completed when they contain 10 messages or they can be completed manually from within DE.MM.DISPLAY. Completed batches are put on the awaiting testkey queue if a testkey is required, with the Msg Disposition of all the messages contained in the batch being set to 'ATK'. Once a testkey has been entered successfully, the batched message is placed on the formatted queue, with the Msg Disposition of all the messages set to 'FORMATTED'. If the carrier is 'TELEX' TELEXP' or 'TELEXF' and batching is not required but a testkey is, the message is placed on the testkey file, with a Msg Disposition of 'ATK'. A Msg Disposition of 'REPAIR' can be changed to Either: 'RESUBMIT' (put on file as 'RESUBMITTED' and another multivalue set created with Msg Disposition of 'RESUBMIT'). Or 'DELETE' (put on file as 'DELETED' because this copy of the message is immediately deleted). Or 'REROUTE' (put on file as 'REROUTED' and another multivalue set created with Msg Disposition of 'REROUTE') 'RESUBMIT' causes a new multivalue set to be created with the same as the message being resubmitted. The Msg Disposition on the new multivalue set is set to 'RESUBMIT' and the original Msg Disposition is set to 'RESUBMITTED'. If the Msg Status or Msg Priority are changed, the new multivalue set will contain the changed values whilst the original Msg Status and Msg Priority will revert to their previous values. The message will then be put on the Unformatted Queue for a second attempt at formatting. This also applies if the Msg Disposition is REROUTE. Messages may be resubmitted not only from the Repair Queue but also when they have been successfully formatted. However, if copies are not allowed (copies on the DE.MESSAGE file is 'NO') for the message type being processed and the Msg Disposition is 'ACK' (acknowledged) or 'WACK' (awaiting acknowledgement), 'REROUTE' and 'RESUBMIT' are not allowed. If a message with a Msg Disposition of 'ACK' or 'WACK' is resubmitted its Msg Disposition becomes 'ACK' - RESUBMITTED' or 'WACK - RESUBMITTED' instead of 'RESUBMITTED'. This subsequent attempt at formatting will not read the Delivery Product file because the Header has already been expanded for the copies required. Disposition 'REROUTE' causes the message to be put on the Unformatted Queue for a second attempt at formatting and also specifies that the Alternate Carrier List should be looked up to override the DE.PRODUCT File information for that copy of the message. If a message with a Msg Disposition of 'ACK' or 'WACK' is rerouted its Msg Disposition becomes 'ACK - REROUTED' or 'WACK - REROUTED' instead of just 'REROUTED'. Messages in the testkey queue and the Batching queue may also have their disposition changed to 'DELETE', 'RESUBMIT' or 'REROUTE'. The message disposition is changed appropriately and the message placed back on the unformatted queue for 'REROUTE' and 'RESUBMIT'. Completed batches of telex messages which have a message removed from them will be re-completed and placed back on the testkey queue. Completed telex batches elsewhere in the delivery system which have messages removed from them are re-completed. The changed batch will then be placed back in the carrier queue it was originally located in or returned to the testkey queue if a testkey is required for it. The individual message removed is then treated in the same way as a resubmitted, rerouted or deleted message. Carrier Control Modules for both SWIFT and Telex pass the message to the Carrier device and set the Msg Disposition to 'WACK' (Awaiting acknowledgement). If the message is successfully received by the recipient (the carrier device receives an acknowledgment), the Msg Disposition is set to 'ACK' (Acknowledged). Otherwise the carrier may 'time out' the message and Carrier Control will receive a NAK (negative acknowledgement). These two possibilities will cause the following changes to the Msg Disposition: Original AcknowledgedNot Ack WACK ACK REPAIR WACK-RESUBMITTED ACK - RESUBMITTEDRESUBMITTED WACK - REROUTED ACK - REROUTEDREROUTED The Carrier Control Module for printing moves each Message to the printer Spool Queue and immediately sets Msg Disposition to 'ACK'. 'DELETE' prevents any further processing or resubmission of the message. When all copies of a message are acknowledged or deleted the message is no longer visible through DE.MM DISPLAY. Validation Rules: Values that are stored on the file: 'FORMATTED' 'ATK' 'RESUBMITTED' 'BATCHING' 'REROUTED' 'REPAIR' 'DELETED' 'WACK' or 'WACK - RESUBMITTED' or 'WACK - REROUTED' 'ACK' or 'ACK - RESUBMITTED' or 'ACK - REROUTED' Values that may be input: 'DELETE', 'RESUBMIT' or 'REROUTE'

      • getMsgStatus

        public TField getMsgStatus()
        Returns the MsgStatus

        Field description: Specifies whether the Message is being held, or is waiting for a period of time before being transmitted. When the Msg Status is 'HOLD' with no time specified, the Message will stay on the Hold Queue indefinitely until the Msg Status is changed to 'RELEASE', then it will be moved to the appropriate Formatted Queue. When 'HOLD' is specified with a time it can still be changed to 'RELEASE' (as above), or left until the stated time of day, when it will be taken from the Hold Queue by the Phantom job MM Timecheck, and placed in the appropriate Formatted Queue. When 'WAIT' is specified the Msg Status is placed on file as 'HOLD hh:mm'. The waiting period entered is added to the current time and the message will be held until this time of day.         Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Describes the type of processing to be done on this copy of the message. Msg Disposition is not present until after formatting when the Header record is expanded to include a multivalue set of data for each copy of the message. If a problem prevents the formatting of a copy of a message then Msg Disposition in the appropriate multivalue set is set to 'Repair', otherwise it is set to 'Formatted'. Once a Message has been Formatted each copy of the Message is put on the Formatted Queue appropriate to the Carrier. However, if the carrier is 'TELEX', 'TELEXP' or 'TELEXF' and the message is of normal priority, the message will be placed in the batching queue if batching is required (Batching Req on DE.PARM is 'Y') and the message type is to be batched (exists on DE.BATCH.GROUP). Messages to be batched will be placed on the batching file with a Msg Disposition of 'BATCHING' until the Batch is complete. Batches will be automatically completed when they contain 10 messages or they can be completed manually from within DE.MM.DISPLAY. Completed batches are put on the awaiting testkey queue if a testkey is required, with the Msg Disposition of all the messages contained in the batch being set to 'ATK'. Once a testkey has been entered successfully, the batched message is placed on the formatted queue, with the Msg Disposition of all the messages set to 'FORMATTED'. If the carrier is 'TELEX' TELEXP' or 'TELEXF' and batching is not required but a testkey is, the message is placed on the testkey file, with a Msg Disposition of 'ATK'. A Msg Disposition of 'REPAIR' can be changed to Either: 'RESUBMIT' (put on file as 'RESUBMITTED' and another multivalue set created with Msg Disposition of 'RESUBMIT'). Or 'DELETE' (put on file as 'DELETED' because this copy of the message is immediately deleted). Or 'REROUTE' (put on file as 'REROUTED' and another multivalue set created with Msg Disposition of 'REROUTE') 'RESUBMIT' causes a new multivalue set to be created with the same as the message being resubmitted. The Msg Disposition on the new multivalue set is set to 'RESUBMIT' and the original Msg Disposition is set to 'RESUBMITTED'. If the Msg Status or Msg Priority are changed, the new multivalue set will contain the changed values whilst the original Msg Status and Msg Priority will revert to their previous values. The message will then be put on the Unformatted Queue for a second attempt at formatting. This also applies if the Msg Disposition is REROUTE. Messages may be resubmitted not only from the Repair Queue but also when they have been successfully formatted. However, if copies are not allowed (copies on the DE.MESSAGE file is 'NO') for the message type being processed and the Msg Disposition is 'ACK' (acknowledged) or 'WACK' (awaiting acknowledgement), 'REROUTE' and 'RESUBMIT' are not allowed. If a message with a Msg Disposition of 'ACK' or 'WACK' is resubmitted its Msg Disposition becomes 'ACK' - RESUBMITTED' or 'WACK - RESUBMITTED' instead of 'RESUBMITTED'. This subsequent attempt at formatting will not read the Delivery Product file because the Header has already been expanded for the copies required. Disposition 'REROUTE' causes the message to be put on the Unformatted Queue for a second attempt at formatting and also specifies that the Alternate Carrier List should be looked up to override the DE.PRODUCT File information for that copy of the message. If a message with a Msg Disposition of 'ACK' or 'WACK' is rerouted its Msg Disposition becomes 'ACK - REROUTED' or 'WACK - REROUTED' instead of just 'REROUTED'. Messages in the testkey queue and the Batching queue may also have their disposition changed to 'DELETE', 'RESUBMIT' or 'REROUTE'. The message disposition is changed appropriately and the message placed back on the unformatted queue for 'REROUTE' and 'RESUBMIT'. Completed batches of telex messages which have a message removed from them will be re-completed and placed back on the testkey queue. Completed telex batches elsewhere in the delivery system which have messages removed from them are re-completed. The changed batch will then be placed back in the carrier queue it was originally located in or returned to the testkey queue if a testkey is required for it. The individual message removed is then treated in the same way as a resubmitted, rerouted or deleted message. Carrier Control Modules for both SWIFT and Telex pass the message to the Carrier device and set the Msg Disposition to 'WACK' (Awaiting acknowledgement). If the message is successfully received by the recipient (the carrier device receives an acknowledgment), the Msg Disposition is set to 'ACK' (Acknowledged). Otherwise the carrier may 'time out' the message and Carrier Control will receive a NAK (negative acknowledgement). These two possibilities will cause the following changes to the Msg Disposition: Original AcknowledgedNot Ack WACK ACK REPAIR WACK-RESUBMITTED ACK - RESUBMITTEDRESUBMITTED WACK - REROUTED ACK - REROUTEDREROUTED The Carrier Control Module for printing moves each Message to the printer Spool Queue and immediately sets Msg Disposition to 'ACK'. 'DELETE' prevents any further processing or resubmission of the message. When all copies of a message are acknowledged or deleted the message is no longer visible through DE.MM DISPLAY.         Field description: Specifies whether the Message is being held, or is waiting for a period of time before being transmitted. If the Header record has not been expanded, status must be changed using Status.
