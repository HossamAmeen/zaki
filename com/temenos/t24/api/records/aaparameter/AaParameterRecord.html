      • getActivityIntegrityCheck

        public TField getActivityIntegrityCheck()
        Returns the ActivityIntegrityCheck

        Field description: Indicates whether system needs to run ‘AA Integrity checks’ when an activity is performed. When this field is flagged, ALL activities performed on ALL arrangements would be checked for integrity during run time of that activity itself(rather than being found in a service later on in COB). It is imperative that as part of integrity, system performs additional checks and when it finds any abnormality, it records the details as part of AA.EXCEPTION.LOG. The default setting is NULL which means no checks and has to be forcefully switched on if the checks have to be done.

        Returns:

      • getAgentBulkNo

        public TField getAgentBulkNo()
        Returns the AgentBulkNo

        Field description: Indicates the count for agent charge activities to be clubbed into a single bill. Based on this field, system will create a charge bill at agent side as consolidated. For Example, If this field is set as 50 then upon reaching 50 charge activities from financial side system agent system will post the consolidated bill for these 50 charges at agent side. This consolidation will happen based on the agent per commission property for the same day.

      • getArchiveActivities

        public TField getArchiveActivities()
        Returns the ArchiveActivities

        Field description: This field denotes the threshold value for activities to be moved from LIVE file to AA.ACTIVITY.HISTORY.HIST table. If the value stated is say 50, when system crosses the 50th activity, it marks the table for archiving. During COB, when the archiving process runs as part of AA.ARCHIVE.SERVICE, (this could be run as online service as well – service name AA.AH.ARCHIVE.SERVICE), system moves the 50 activity multi-values to AA.ACTIVITY.HISTORY.HIST table. When backdated activities are performed prior to this date, system automatically restores the content from HIST file where relevant and when the processing is complete, again marks the arrangement for archival which would be picked in COB process or online service.

        Parameters:
        Returns:
        TField - The ArchiveActivities

      • getArchiveBills

        public TField getArchiveBills()
        Returns the ArchiveBills

        Field description: This field denotes the threshold value for settled bills to be moved from LIVE file to AA.BILL.DETAILS.HIST table. If the value stated is say 20, when system creates the 21st bill, it marks the table for archiving. During COB, when the archiving process runs(this could be run online as well – service name AA.BILL.ARCHIVE.SERVICE), system moves the 20 bills to AA.BILL.DETAILS.HIST table. When moving so, it also removes the content from the live AA.ACCOUNT.DETAILS of these bill references.

      • getBankGoLiveStatus

        public TField getBankGoLiveStatus()
        Returns the BankGoLiveStatus

        Field description: This field indicates whether the bank is in the Live or Implementation phase. Allowed value is LIVE or can be left blank LIVE - Indicates that the bank is live. Blank - Indicates bank is in the implementation stage. If the Bank is in the implementation stage, it is prudent to identify abnormalities arising out of the lifecycle scenarios tested. Any abnormalities found at this stage will result in an error so that these are reported upfront. However, the Post Live stage, it refers to the ACTIVITY.INTEGRITY.CHECK field and decides if any abnormalities should be stopped and reported or should be logged in AA.EXCEPTION.LOG while allowing the transaction to continue.

        public void setArchiveActivities(java.lang.CharSequence value)

        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: This field denotes the threshold value for activities to be moved from LIVE file to AA.ACTIVITY.HISTORY.HIST table. If the value stated is say 50, when system crosses the 50th activity, it marks the table for archiving. During COB, when the archiving process runs as part of AA.ARCHIVE.SERVICE, (this could be run as online service as well – service name AA.AH.ARCHIVE.SERVICE), system moves the 50 activity multi-values to AA.ACTIVITY.HISTORY.HIST table. When backdated activities are performed prior to this date, system automatically restores the content from HIST file where relevant and when the processing is complete, again marks the arrangement for archival which would be picked in COB process or online service.
