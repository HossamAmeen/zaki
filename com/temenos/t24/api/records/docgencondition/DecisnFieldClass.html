      • getAscCusFld

        public TField getAscCusFld()
        Returns the AscCusFld

        Field description: Name of the Customer field in the application, associated with the decision field (DECISN.FIELD). Example: In DOC.GEN.CONDITION for SEC.TRADE, if the DECISN.FIELD is CUSTOMER.RESIDENCE which is a multi-valued field returning one value for each Customer in the transaction, then the value of ASC.CUS.FLD could be set to CUSTOMER.NO. When Documents are tracked for each Customer (CUSTOMER.NO), then the value of CUSTOMER.RESIDENCE corresponding to that Customer would be used for comparison purposes. Validation Rules: Input not allowed when the field DECISN.FIELD in the set is Single-Valued. Input optional when the field DECISN.FIELD in the set is Multi-Valued. If the DECISN.FIELD is multi-valued and no ASC.CUS.FLD value is specified, then the first multi-value of the DECISN.FIELD would be used for comparison purposes. The field specified here should be Multi-Valued.

        Returns:

      • getDecisnField

        public TField getDecisnField()
        Returns the DecisnField

        Field description: Can be either a local subroutine name or a field defined in the STANDARD.SELECTION record for the application (identified by the first part of the key field). If a local subroutine is defined then the subroutine must be prefixed with the @ character. Input for a field defined on STANDARD.SELECTION can be either the field number or the field name. LOCAL SUBROUTINE (Technical Details): The local subroutine must be defined on the PGM.FILE as a V type program. Example: PROGRAM FILE SEE PROGRAM DM.TEST.ROUTINE ---------------------------------------------------------------------- 1 TYPE.............. V 2. 1 GB SCREEN.TITLE DM.TEST.ROUTINE 3 ADDITIONAL.INFO... .PAR() 5 PRODUCT........... DM The local subroutine is called from the T24 subroutine DM.GRP.CONDITION which would determine the relevant group(s) the contract falls (if any). The local subroutine must be defined to accept the following arguments: APPLN: Name of the application ID.APPLN: ID of the application record for which documents are tracked R.APPLN: The application record itself FIELD.VALUE: Value of the field specified as parameter in the routine attached to DOC.GEN.CONDITION. If the parameter specified is multi-valued field and a value is also specified for ASC.CUS.FLD, then this value would be the value of the parameter field corresponding to the value of ASC.CUS.FLD. COMPARISON: Value of DECISION in DOC.GEN.CONDITION DATA.FROM: Value of DECISN.FROM in DOC.GEN.CONDITION DATA.TO: Value of DECISN.TO in DOC.GEN.CONDITION CONDITION.OK: Value returned should be null or 0 for False or any other value for True. Sample routine to test OR condition when the values in DECISN.FROM are separated by Spaces: 0001: SUBROUTINE DM.TEST.ROUTINE(APPLN,ID.APPLN,R.APPLN,FIELD.VALUE,COMPA RISON,DATA.FROM,DATA.TO,CONDITION.OK) 0002: 0003: $INSERT I_COMMON 0004: $INSERT I_EQUATE 0005: 0006: IF COMPARISON EQ "EQ" THEN 0007: DECS.FROM = DATA.FROM 0008: CONVERT " " TO FM IN DECS.FROM 0009: LOCATE FIELD.VALUE IN DECS.FROM<1> SETTING CONDITION.OK ELSE CON DITION.OK = 0 0010: END 0011: RETURN 0012: 0013: END When the routine is specified as a DECISN.FIELD in DOC.GEN.CONDITION, any field name in the Application specified in the ID of the DOC.GEN.CONDITION could be specified as its parameter. Example of a DOC.GEN.CONDITION record for FT, where the above ref. sample routine is attached to test the OR condition for the DEBIT.ACCT.NO field. The values to be tested are specified in DECISN.FROM separated by spaces. DOC.GEN.CONDITION DOC.GEN.COND.ID... FT ------------------------------------------------------------------------------ 1 SCAN.ALL.GROUP.... NO 2. 1 DOC.GROUP...... 1 3. 1. 1 DECISN.FIELD @DM.TEST.ROUTINE(DEBIT.ACCT.NO) 4. 1. 1 DECISION.... EQ 5. 1. 1 DECISN.FROM. 19038 19108 Validation Rules: Optional input. Could be either a valid field name specified in the STANDARD.SELECTION of the application defined in the ID of the record or a local sub-routine name pre-fixed with @. @ID cannot be given as a Decision Field. If a routine is attached, it should be specified as a V type program in PGM.FILE. The parameter of the attached routine should be a valid field name specified in the STANDARD.SELECTION of the application defined in the ID of the record.

        Parameters:
        Returns:
