      • getApplication

        public TField getApplication()
        Returns the Application

        Field description: Indicates the Application for which the start point for Auto numbering is to be loaded. This can optionally be suffixed by a department account officer code. If a DAO code is specified then the start point entered in the field ID.START is used only for users with that DAO code. This allows departments to have different starting ranges for transaction ids. Each day the first id allocated, when using [F3]/next id, will be the Application:JulianDate:ID.START. e.g: An AUTO.ID.START records with the following information: APPLICATION : SWAP-100 ID.START : SW9500101000 APPLICATION : SWAP-200 ID.START : SW9500102000 APPLICATION : SWAP ID.START : SW9500100001 This means all users in Department 100 will create SWAP contracts starting from SWyyddd01000 and all user in Department 200 will create SWAP contracts staring from SWyyddd02000. All other users will start from SWyyddd00001. Validation Rules: Up to 32 alphanumeric characters, type 'A'. Multivalue field associated with ID.START. Mandatory input. Must be a valid APPLICATION with an optional DEPT.ACCT.OFFICER in the format: APPLICATION-DEPT.ACCT.OFFICER The Application name must be present on the PGM.FILE with a type of H, U or W. No duplications allowed.

      • getIdLength

        public TField getIdLength()
        Returns the IdLength

        Field description: Used in conjunction with UNIQUE.NO to restrict the maximum length of any generated id. Should be used with caution as the underlying application may expect id lengths to be of a certain definition and length. If settings in UNIQUE.NO and BASE.TABLE are made it may be advisable to restrict the length of the id to the standard format of 12, 14 or 16 depending on the application. If left blank the UNIQUE.NO & BASE.TABLE combination may extend the id length to provide sufficient unique id's. It is advisable to contact Temenos before setting this field on T24 applications.

      • getIdStart

        public TField getIdStart()
        Returns the IdStart

        Field description: Is the key of the Application file from which auto-numbering is to start. This field contains the key, from which auto numbering is to start. The details entered must be in the correct format for the file concerned. There are no checks that the data has been entered in the correct format. Where the key contains a date the date entered must be less than the current run date, for auto numbering to start correctly. The incremental number will be reset each working day as the julian date is contained within the key. The format of the keys for the 'Contract' Application like Foreign Exchange and Money Market are as follows: 4 digit - Optional Sub-Application Identifier (used within the Securities Application) 2 digit - Application 5 digit - Date in the format YYDDD 5 digit - Sequence number from 00001 to 99999 The format for each application must be obtained before entering data. The sequence number entered here is used as the starting point for the next days contract ids. Validation Rules: Up to 35 alphanumeric characters. Multivalue field associated with APPLICATION. Mandatory field.

        Returns:
        TField - The IdStart

      • getUniqueNo

        public TField getUniqueNo()
        Returns the UniqueNo

        Field description: If set this will invoke the generation of transaction id's based on system generated values. A record will be written to the LOCKING file with a name like application.UNIQUE (application such as REPO would appear as REPO.UNIQUE). The presence of such a record will trigger the auto generation of sequences using default characters (1-9 and B-Z - excluding vowels). Dependent on the use of the BASE.TABLE and ID.LENGTH fields, the sequence generated will contain alpha, numeric or both in the key, subject to the length allowed. The benefit of this is that there is no exclusive lock placed on the LOCKING file when the next available sequence is requested. The system will generate a value based on the criteria set and check if it is used, it wil retry until a unique value is found. With the correct settings a unique id will be found each time. Validation Rules: Optional input Valid input is 'Yes' or 'Null'

        Returns:
