        public void addProductNo(java.lang.CharSequence param)
        Appends the specified ProductNo to the end of the collection.
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Specifies the Limit product (ie LIMIT.REFERENCE number) for which the user wants to define a Limit control within the Application entered in the previous field. For all Financial Applications (eg Loans, Money Market, Accounts, Foreign Exchange etc), T24 allows the user to define the Limit products according to his own requirements. The definition of these various Limit products will be done by Application using the association of the multivalue Fields 13 to 18. Whenever Market conditions change or the Bank's management decides to reorganise their Limit control, a simple maintenance in this table will automatically allow and take care of such changes in the Limit verification process. Validation Rules: 3 to 4 numeric characters (or) alpha-numeric value with the maximum length 12 Subvalue field. Input mandatory when the field APPLICATION has been defined; otherwise not allowed. No default value The PRODUCT.NO must be a valid ID on the LIMIT.REFERENCE file. Input in this field will only be allowed when the APPLICATION has been defined in Field 13.1. This table will allow the input of a valid product number eg 100,200,LR12345,SHORTTERM ... 9600 or a valid sub-product number eg 101, 102, 103 , LR12345 , SHORTTERM etc. Definition of product together with sub-product will not be allowed. Either the user chooses to enter 101,102, 103 etc... or, 100, 200 ... 9600. The product or sub-product number will be checked in connection with application and field FX.OR.TIME. BAND in the LIMIT.REFERENCE table as follows: . The application 'FOREX' must have the field FX.OR.TIME. BAND' with a value = to 'FX'. . The application 'ACCOUNT' must have the field 'FX.OR.TIME. BAND' with a value of 'TB' - For Arrangement Lending contracts i.e. Time band possible or NULL - For accounts created using Account module and/or Arrangement account product . The application FD.FIDUCIARY must have the field FX.OR.TIME.BAND with a value of 'IN' or 'TB'. - The application FD.FID.ORDER must have the field 'FX.OR.TIME.BAND' with a value of 'DP' . Other Applications must have the field 'FX.OR.TIME.BAND' with a value equal to 'TB' (i.e. Time band possible) or null. For Limit References in a Cross Limit structure, only the SUB-PRODUCT Limit References (value of CROSS.LIMIT is PROD in the LIMIT.REFERENCE record) would be allowed.

        Parameters:
        param - the ProductNo to append
        Returns:

      • getApplication

        public TField getApplication()
        Returns the Application

        Field description: Specifies the Application for which the user wants to create specific Limit products. Only Financial Type Applications will be allowed in this field. For all Financial Applications (eg Loans, Money Market, Accounts, Foreign Exchange etc), T24 allows the user to define the Limit products according to his own requirements. The definition of these various Limit products will be done by Application using the association of the multivalue Fields 13 to 18. Whenever Market conditions change or the Bank's management decides to reorganise their Limit control, a simple maintenance in this table will automatically allow and take care of such changes in the Limit verification process. It is also possible to specify a suffix after the application separated by a "-" character. This will allow sub-products to be defined for a particular application, so that more than 1 limit refererence can be defined for an application. For example the application SL.SYNDICATED.LOANS may require 2 limit references, for the underwritten portion and the agent participation. The following could be defined: APPLICATION SL.SYNDICATED.LOANS PRODUCT NO 4000 This is the overall for the SL application and will default 4000 as the limit reference APPLICATION SL.SYNDICATED.LOANS-UNDERWRITE PRODUCT.NO 4100 The reference 4100 will be used for undewritten limits APPLICATION SL.SYNDICATED.LOANS-AGENT PRODUCT.NO 4200 The reference 4200 will be used for the AGENT portion. The order for defaulting is as follows: 1 APPLICATION-suffix with specific conditions 2 APPLICATION-suffix default 3 APPLICATION with specific conditions 4 APPLICATION default The suffix used by applications when defaulting limit references is hard coded if required. To find if an application requires a suffix consult the application documentation. Validation Rules: Any valid Application name (1 to 25 characters) where the PGM.TYPE has a suffic equal to '.NOH' ie that it represents a 'Financial' application. Multivalue field associated with Fields 14 to 18. The Application entered in this field must be a valid Application in PGM.FILE. The PGM.TYPE of the Application (except ACCOUNT) must have a suffix of '.NOH' ie it must relate to an Application with financial figures. At least a 'Default' definition will be mandatory for each Application entered in this multivalue field. The default of an Application will be recognised by the fact that the PRODUCT NO has been entered in Field 14.1 while the Fields 15.1.1 to 18.1.1 have been left blank. Duplicates of the same Application name will only be allowed when the fields 'DECIS FIELD' to 'DECISION TO ' have been defined with different values. A suffix may be supplied separated by a "-" character. When supplied a default condition must exist for the suffix, as well as a definition for the application without suffix as the overall default. If Limit Parameter record ID exists for an application name, then this name is not allowed in the 'APPLICATION' field of Limit Parameter SYSTEM record

        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Specifies the Application for which the user wants to create specific Limit products. Only Financial Type Applications will be allowed in this field. For all Financial Applications (eg Loans, Money Market, Accounts, Foreign Exchange etc), T24 allows the user to define the Limit products according to his own requirements. The definition of these various Limit products will be done by Application using the association of the multivalue Fields 13 to 18. Whenever Market conditions change or the Bank's management decides to reorganise their Limit control, a simple maintenance in this table will automatically allow and take care of such changes in the Limit verification process. It is also possible to specify a suffix after the application separated by a "-" character. This will allow sub-products to be defined for a particular application, so that more than 1 limit refererence can be defined for an application. For example the application SL.SYNDICATED.LOANS may require 2 limit references, for the underwritten portion and the agent participation. The following could be defined: APPLICATION SL.SYNDICATED.LOANS PRODUCT NO 4000 This is the overall for the SL application and will default 4000 as the limit reference APPLICATION SL.SYNDICATED.LOANS-UNDERWRITE PRODUCT.NO 4100 The reference 4100 will be used for undewritten limits APPLICATION SL.SYNDICATED.LOANS-AGENT PRODUCT.NO 4200 The reference 4200 will be used for the AGENT portion. The order for defaulting is as follows: 1 APPLICATION-suffix with specific conditions 2 APPLICATION-suffix default 3 APPLICATION with specific conditions 4 APPLICATION default The suffix used by applications when defaulting limit references is hard coded if required.
