      • getAdvancePaymentMethod

        public TField getAdvancePaymentMethod()
        Returns the AdvancePaymentMethod

        Field description: This field denotes how the repaid amount should be applied to produce the future scheduled bills in advance. This field will hold either one of the 2 values PARTIAL,FULL and they are mandatory for the ADVANCE Payment rule type.

        Example On 28th Dec, a repayment is made for $900 which can settle 2 future CONSTANT schedules each of $500 on 02-Jan fully and 07-Jan partially If the repayment is 100 against the current interest if accrued (ACCLOANINTEREST) and the rest 800 could be allocated against the principal (CURPRINCIPAL). FULL - Denotes the payment rules is applicable only if the amount can fully settle future scheduled bills. System will raise "SETTLED" bills for these future schedules with full repayment date as the financial date System will raise bills only if the repayment settles the entire bill amount. In this case, system will generate and settle bill for 02-Jan while remaining $400 will be processed by its remainder activity. 02 Jan - Bill 1 - $500 will be OR.TOTAL.AMOUNT and OS.TOTAL.AMOUNT will be 0 and the Bill date, payment date will be 02 Jan and the financial date will be 28th Dec, Bill statuses are Advanced, Settled

        PARTIAL - Denotes the repaid amount is eligible to settle future scheduled bills even if it settles only partial amount. System will raise an "ADVANCED" bill for this schedule with original financial date as the financial date Example - PARTIAL For the same example provided above, system will raise a bill in addition to '02 Jan' Bill, as the amount could also settle a future schedule in partial 07 Jan - Bill 2 - $400 will be allocated fully towards the ACCOUNT, and INTEREST portion will be 0 . In Bill the OR.TOTAL.AMOUNT will be 500, with financial date as 07th Jan and then system will update the Bill status as Advanced. 07 Jan - Bill 2 - The remaining $100 will be made due on its original payment date against the principal amount and update the bill status as "DUE"

        Returns:
        TField - The AdvancePaymentMethod

      • getApplicationType

        public TField getApplicationType()
        Returns the ApplicationType

        Field description: Linked to AA.PAYMENT.RULE.TYPE This field is used to specify how the payment rules will apply to an arrangement.

        This can have any of the below three values.

        Current: Payment will be applied on arrangement's current balances. (e.g. current principal, accrued interest, etc.). This will settle the current balances and doesn’t look at application order. Input to application order will not be allowed for Current Balance repayments.

        Bill Property: For product with bills and due amounts, payment to the due amounts will be Property wise(e.g. all Principal amounts followed by all Interest amounts, etc.)

        Bill Date: For product with bills and due amounts, payment to the due amounts will be bill date wise (e.g., all amount on bill 1 followed by all amounts of bill 2, etc.).

        Ex1: Bill Property

        Number of Bills : Bill1 Bill2 Bill3 Property in each Bill : Bill1prop1, Bill1prop2, Bill2prop1, Bill2prop2, Bill3prop1, Bill3prop2

        Order of settling if Application Order is Oldest First Bill1prop1, Bill2prop1, Bill3prop1, Bill1prop2, Bill2prop2, Bill3prop2

        Order of settling if Application Order is Oldest Last Bill3prop1, Bill2prop1, Bill1prop1, Bill3prop2, Bill2prop2, Bill1prop2

        Ex 2: Bill Date

        Number of Bills : Bill1 Bill2 Bill3 Property in each Bill : Bill1prop1, Bill1prop2, Bill2prop1, Bill2prop2, Bill3prop1, Bill3prop2

        Order of settling if Application Order is Oldest First Bill1prop1, Bill1prop2, Bill2prop1, Bill2prop2, Bill3prop1, Bill3prop2

        Order of settling if Application Order is Oldest Last Bill3prop1, Bill3prop2, Bill2prop1, Bill2prop2, Bill1prop1, Bill1prop2

        Parameters:
        Returns:
        TField - The ApplicationType

      • getFinancialStatus

        public TField getFinancialStatus()
        Returns the FinancialStatus

        Field description: This field allows different payment allocation rules to be setup for performing assets and non-performing assets, if cad accept the following options,

        1. PERFORMING: If this option is chosen, the rules stated under this multi-value would be used when the repayment happens on the contract when the contract is not suspended. Just like normal repayment of a DUE bill.

        2. SUSPENDED: If this option is chosen, the rules would be applicable only when the contract is under suspension. The allocation themselves might be different – but the working of the payment rules component as such does not change.

        3. BOTH: If this option is chosen, system would use the same rule stated under it for both Suspension and performing loans(indicating, the bank does not want to differentiate its repayment between performing or suspended loans). Since we also need to worry about the existing clients who are using Payment rules and upgrade to this release, a conversion would be written and BOTH would be flagged by "default" during these conversions.

        4. RESTRUCTURE: If this option is chosen, irrespective of performing or non-performing status, rules stated under this multi-value will be used if the contract is restructured.

        Validation Rules

        Mandatory field

        Returns:

      • getMakeBillDue

        public TField getMakeBillDue()
        Returns the MakeBillDue

        Field description: In the case where billed amounts exist but the due date has not yet been reached, users can determine that the billed amounts can be made due up to the amount of any remaining payment. This allows for the processing of bill payments which are received prior to the due date.

        It can accept either YES or NO.

        Ex:

        Repayrule 1:

        Application Order – Oldest Last Application Type – Bill Date Sequence – 1, 2 Property – Loaninterest, Loanaccount. Make Bill Due – Yes Remainder Activity - LENDING-APPLYPAYMENT-REPAYRULE2

        Repayrule 2:

        Application Order – Oldest Last Application Type – Bill Date Sequence – 1, 2 Property – Prinincreasecharge, Disbursecharge Remainder Activity – LENDING-APPLYPAYMENT-CURRENT

        Bill Details

        Bill 1 – Scheduled Bill Bill Amount - 10000 Issue Date = 10 Jan 2008 Due Date = 10 Jan 2008

        Bill 2 – Activity Charge bill raised because of Principal Increase Bill Amount - 1000 Issue Date = 1 Jan 2008 Due Date = 20 Jan 2008

        Bill 3 – Activity Charge bill raised because of Disbursement Bill Amount - 1500 Issue Date – 5 Jan 2008 Due Date – 20 Jan 2008

        Repayment Amount – 12500 Repayment Date – 15 Jan 2008

        Order of settling

        Repayrule1 will settle 10000 for Bill1 and will DUE the activity charge bills so that repayrule2 can settle bills related to activity charges

        Parameters:
        Returns:
        TField - The MakeBillDue

      • getRemainderActivity

        public TField getRemainderActivity()
        Returns the RemainderActivity

        Field description: When a payment is made which satisfies all of the due and overdue amounts and there is a remainder of funds (i.e. an overpayment), then this field can be used to handle remainder of funds.

        This is handled by specifying the Activity to be processed for the remainder of the payment. This will be subject to any Transaction Rules. Should the transaction fail, the remainder amount will be processed by the Default Activity specified in Accounting.

        Ex 1:

        Bill Due Amount – 10000 Remainder Activity – LENDING-APPLYPAYMENT-CURRENT Repayment Amount – 15000

        Settlement Method

        Bill due amount will be settled with 10000 and the remaining 5000 will be used for settling current balances

        Returns:

        Field description: When a payment is made which satisfies all of the due and overdue amounts and there is a remainder of funds (i.e. an overpayment), then this field can be used to handle remainder of funds.

        This is handled by specifying the Activity to be processed for the remainder of the payment. This will be subject to any Transaction Rules. Should the transaction fail, the remainder amount will be processed by the Default Activity specified in Accounting.

        Ex 1:

        Bill Due Amount – 10000 Remainder Activity – LENDING-APPLYPAYMENT-CURRENT Repayment Amount – 15000

        Settlement Method

        Bill due amount will be settled with 10000 and the remaining 5000 will be used for settling current balances
