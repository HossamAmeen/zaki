      • getCrFloorLmt

        public TField getCrFloorLmt()
        Returns the CrFloorLmt

        Field description: This field specifies the minimum value (transaction amount) to be reported on the requested message and same is mapped to tag 34F (Floor Limit) in outward MT920. The amount entered here along with the Currency given in the previous field is mapped to tag 34F for outward MT920 with D/C Marker as as C subject to the following:- 1. When DR.FLOOR.LMT is given as 10000.00, AC.CURRENCY as "USD" & this field is left blank ,then the outward MT920-tag 34F appears as USD10000,00- which indicates all debit or credit transaction exceeds USD10000 has to be reported in the Interim statement. 2. When DR.FLOOR.LMT is given as 10000.00, CR.FLOOR.LMT is given as 15000.00 & AC.CURRENCY as "USD", then in outward MT920-first tag 34F appears as USDD10000,00 & in second tag 34F appears as USDC15000,00 which indicates debit entries in excess of USD10000 & Credit entries in excess of USD15000 has to be reported in the interim statement. Validation Rules: Optional field -When Message.Type is specified as 942 and when XML.STMT.TYPE is defined Input allowed only when DR.FLOOR.LMT is entered with different Amount. 1-14 characters plus a decimal point (Standard Amount format) - type AMT.

        Parameters:
        Returns:

      • getInAccount

        public TField getInAccount()
        Returns the InAccount

        Field description: This field identifies the Account for which transaction information requested via incoming MT920. The tag 25 (Account Identification) detail in MT920 message, is mapped here during Inward processing. The Incoming Account available here should be a valid T24 Account-otherwise Error raised. There should be a Bilateral agreement record in existence as on the creation date for the combination of this Account Customer, the Sender of MT920 ( as populated in the previous field) and for the Message type-Otherwise Error Raised. In addition to the above, to generate MT941,MT942 messages, in response to Incoming MT920, the Send.Msg.Type should be set to "STP" in ACCOUNT.STATEMENT application for the Account mentioned here-Otherwise Error raised. Valid Account number can be inputted here incase the T24 Account number cannot be derived through the Incoming MT920 process-However other incoming details cannot be modified. Validation Rules: Updated by System with the Incoming MT920 -Tag 25 -Account Number Can be changed to a Valid T24 Account Number before authorisataion for direct Inward No input field for direction - Outward

        Parameters:
        Returns:
        TField - The InAccount

      • getMessageType

        public TField getMessageType()
        Returns the MessageType

        Field description: For outward instruction, specify the message type which is to be requested and same is mapped to tag 12 (Message Request) of MT920. Once an inward MT920 is processed, the tag 12 details is mapped in this field. The fields Message.Type to Cr.Floor.Lmt are Associated multi value sets and same is mapped to outward MT920 for each repeating sequence. For Inward MT920, for each repeating sequence a record in DE.STATEMENT.REQUEST is created. Please note the incoming MT920 details mapped to this application cannot be modified except for the Incoming Account Number. Duplicate values for the combination of Out.Account & Message.type is not allowed. Validation Rules: Mandatory field if XML.STMT.TYPE is not specified Valid values are "940"- Customer Statement, "941"-Balance Report, "942"- Interim Transaction Report,"950"-Statement Message

        Parameters:
        Returns:
        TField - The MessageType
