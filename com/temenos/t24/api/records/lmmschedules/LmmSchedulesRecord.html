        Field description: This field contains the value, and therefore Schedule, date of the financial movement to which the 'S' Schedule relates. When the End of Day processes a type 'S' Schedule it needs to know what components need to be paid. This field contains the julian date of the financial schedule to which this 'S' Schedule relates. It is the financial schedule that specifies the components that are to be paid. Example: A deposit maturing on 11th March 1997 will have a schedule as follows: Key =Contract Number/97070/00 (11th March) Schedule Types = P and I The trigger to check whether a payment needs to be sent for this Contract will appear on 10th March (given one day delivery) and will appear as follows: Key = Contract Number/97069/00 (10th March) Schedule Types = S S Date = 1997070 (the 11th March) Validation Rules: Seven digit julian date where the first four digits are year (e.g. 1997) and the last three digits are day number (e.g. 001 is January 1st).

      • setTypeS

        public void setTypeS(TField value)
        Set the specified value for TypeS
        Throws an UnsupportedOperationException if the object is not modifiable.

        Field description: Specifies whether there could be a payment instruction to send on this Schedule date or not. If during the future life of a Contract, funds have to be delivered to the Customer then this field will Contain a 'Y' 'n' days prior to the value date of the movement of funds. The number of days prior to the movement of funds value date is determined by the DAYS.DELIVERY in the payment CURRENCY record. This is best explained by means of an example: If a U.S. dollar deposit is due to mature on 11th March 1986 then a 'Financial Schedule' Will be written for the 11th March that includes type 'P' for Principal repayment, and type 'I' for interest payment. If the U.S. dollar currency record states that there should be '1' day delivery then any physical payment instruction that needs to be sent must be sent one day prior to the payment value date. In the example above this means that should a payment instruction need to be sent to repay the deposit, then it needs to be sent on the 10th March. In order to 'trigger' this physical sending of a payment message, the system writes a Schedule 'type S' 'n' days prior to any movement of funds out of the bank. The value for 'n' is derived from the days delivery field in the payment currency record and so continuing with the example above, a Schedule 'type S' will be written for 10th March. It should be noted that at the time of writing the Schedules, the system does not determine whether a physical payment instruction is required or not - it merely establishes the 'trigger'. When the 'End of Day' processes the 'S' schedule then it reviews the settlement instructions and either writes a payment entry (which usually results in the sending of a payment instruction) or it does nothing if the account to credit is an internal or current account. To determine which financial movement any 'S' Schedule relates to, the next field 'S DATE', is completed with the schedule date of the financial movements. Validation Rules: Only values are 'Y' meaning yes there could be a payment to send; or blank meaning there is not.
